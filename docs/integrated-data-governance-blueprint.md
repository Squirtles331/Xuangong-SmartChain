# 玄工智链一体化数据治理与主数据蓝图

## 1. 文档定位

本文档用于回答一个当前阶段非常关键的问题：

- 玄工智链已经完成了 `PLM / MES / WMS / ERP / SRM / QMS / APS / BI / MDM / SYS` 的业务域拆分
- 但系统还没有形成真正的一体化数据底座
- 后续如果继续只铺静态页面、不先统一核心数据对象，最终会再次落入“系统很多、数据不通、流程脱节”的老路

因此，本文档的目标不是讨论抽象的数字化理论，而是给出玄工智链下一阶段可以直接执行的设计基线：

- 哪些内容应纳入统一主数据治理
- 哪些对象由哪个模块主导维护
- 哪些跨系统业务链要优先打通
- 静态页、mock、接口应按什么顺序推进

## 2. 当前系统判断

### 2.1 已经具备的基础

当前系统已经完成业务域拆分，具备继续做一体化设计的结构基础：

- `aps`
- `bi`
- `crm`
- `eam`
- `erp`
- `iot`
- `mdm`
- `mes`
- `plm`
- `qms`
- `srm`
- `sys`
- `wms`

### 2.2 当前最明显的短板

当前 `MDM` 仍然偏薄，主要还是基础资料页集合，而不是一体化主数据中心。现有页面主要包括：

- 物料
- 模具
- 组织
- 资源
- 工作中心

这意味着当前系统虽然有“模块架构”，但还没有“统一对象架构”。如果此时直接进入大量 mock 或接口联调，后续很容易出现：

- PLM、MES、ERP、WMS 各自维护近似但不一致的物料与 BOM 数据
- 工艺、用量、库存、成本口径不统一
- 页面能展示，但业务链无法闭环
- 状态流和数据流脱节，系统退化为多个孤立台账

## 3. 对当前系统最有价值的结论

结合制造企业常见问题与玄工智链当前阶段，最重要的结论有三条：

### 3.1 现在最优先的不是“更多页面”，而是“统一对象”

后续静态页开发、mock 设计、接口设计，都必须围绕统一对象进行，而不是围绕单页面字段进行。

### 3.2 现在最需要补的不是集成平台，而是数据底座

当前阶段应优先完成：

- 主数据对象清单
- 主数据字段口径
- 唯一来源系统
- 下游消费关系
- 跨模块状态约束

而不是过早进入 ESB、iPaaS、事件平台等偏基础设施层面的重投入设计。

### 3.3 MDM 必须从“资料维护”升级为“统一主数据中心”

`MDM` 不应只负责几个基础 CRUD 页，而应承担：

- 主数据唯一源头
- 编码与分类标准
- 多系统统一口径
- 发布与同步基线
- 数据质量治理入口

## 4. 一体化数据治理总体原则

### 4.1 一码一物，同源同义

对物料、单位、仓库、客户、供应商、工作中心等核心对象建立唯一身份，不允许多个模块各自定义不同版本。

### 4.2 主字段唯一来源，下游系统只消费或扩展

所有核心对象必须明确：

- 谁创建
- 谁审批
- 谁发布
- 谁消费
- 谁允许补充执行态字段

### 4.3 页面设计先服从业务链，再服从模块边界

静态页不是为了把模块填满，而是为了还原真实业务链。

### 4.4 先建立约束，再做联动

即使当前阶段暂不开发完整接口联动，也要先在页面模型、状态模型、mock 数据中体现业务约束。

## 5. 统一对象分层

建议将玄工智链对象分成四层：

### 5.1 主数据层

长期稳定、跨系统共享、被频繁引用的对象。

例如：

- 物料
- 单位
- 分类
- 仓库
- 库位
- 组织
- 工作中心
- 资源
- 设备基础档案
- 客户
- 供应商
- 编码规则
- 状态字典

### 5.2 产品与工程定义层

描述“应该如何制造”的对象。

例如：

- EBOM
- MBOM
- 工艺路线
- 工序
- 工艺参数
- 版本
- 图纸
- 设计变更

### 5.3 计划与交易层

描述“准备做什么”和“业务上发生了什么”的对象。

例如：

- 销售订单
- 采购订单
- 生产订单
- 排程结果
- 库存事务
- 收货单
- 发货单

### 5.4 执行与追溯层

描述“现场实际发生了什么”的对象。

例如：

- 派工任务
- 报工记录
- 实际领料
- 实际耗用
- 检验记录
- 异常记录
- 在制品状态
- 批次追溯链

## 6. 建议的模块主责边界

以下不是菜单归属，而是对象主责归属。

| 模块  | 主责对象                                                               | 说明                             |
| ----- | ---------------------------------------------------------------------- | -------------------------------- |
| `MDM` | 物料、单位、分类、组织、工作中心、仓库、库位、基础资源、编码规则、字典 | 统一主数据源，不承载现场执行真相 |
| `PLM` | EBOM、MBOM、工艺路线、版本、图纸、ECN                                  | 定义产品与工程基线，负责变更发布 |
| `ERP` | 销售订单、采购订单、库存账、财务成本、应收应付                         | 负责经营账与资源计划口径         |
| `MES` | 工单、派工、报工、实际耗用、在制、异常、执行追溯                       | 负责生产执行真相                 |
| `WMS` | 收货、上架、拣货、发运、盘点、库存位置事务                             | 负责仓储执行真相                 |
| `QMS` | 检验标准、检验记录、不合格裁决、质量处置                               | 负责质量裁决与质量记录           |
| `SRM` | 供应商协同、送货预约、对账协同                                         | 负责供应链协同门户流程           |
| `APS` | 计划计算、排程结果、约束分析                                           | 负责计划计算结果，不做主数据源   |
| `CRM` | 客户经营、报价、合同、订单线索                                         | 负责客户经营前端对象             |
| `EAM` | 设备维修、保养、点检执行                                               | 负责设备运维执行                 |
| `BI`  | 指标聚合、分析主题、经营看板                                           | 不拥有业务真相，只消费各域数据   |

## 7. 第一阶段必须补齐的主数据范围

这是当前系统最应该优先补齐的一组对象。

### 7.1 物料主数据

至少应包括：

- 物料编码
- 物料名称
- 规格型号
- 物料分类
- 单位
- 批次管理策略
- 序列号策略
- 替代料策略
- 安全库存参数
- 默认仓储属性
- 采购属性
- 生产属性
- 质量属性

### 7.2 BOM 主数据

至少应明确：

- BOM 类型
- 版本
- 生效状态
- 子项物料
- 标准用量
- 损耗率
- 替代关系
- 虚拟件标识
- 发布来源

### 7.3 工艺与制造定义

- 工艺路线
- 工序
- 工作中心映射
- 标准工时
- 标准产能
- 工艺参数
- 工艺版本

### 7.4 仓储主数据

- 仓库
- 库区
- 库位
- 库存状态
- 出入库策略
- 批次维度

### 7.5 供应链与经营基础对象

- 供应商主数据
- 客户主数据

### 7.6 质量主数据

- 检验标准
- 检验项目
- 判定规则
- 不合格代码

## 8. 玄工智链优先打通的三条业务主链

当前阶段不建议一开始就铺全系统，而应优先围绕三条链路做静态页和后续 mock。

### 8.1 产品定义到生产执行链

`物料 -> BOM -> 工艺路线 -> 生产工单 -> 领料 -> 报工 -> 完工 -> 入库`

这是玄工智链的一号主链，直接决定系统是否具备制造中枢能力。

### 8.2 销售到交付链

`客户/报价 -> 销售订单 -> 计划 -> 生产 -> 入库 -> 发货 -> 对账`

这是产销协同链，决定系统是否能支撑订单履约。

### 8.3 采购到来料入库链

`供应商 -> 采购订单 -> 收货 -> 检验 -> 入库 -> 领用`

这是供应链协同链，决定库存与质量口径是否一致。

## 9. 必须提前固化的跨模块业务规则

即使当前阶段只做静态页，也应先把这些规则固化在页面模型和 mock 设计中：

- 销售订单未审核，不允许下达正式生产任务
- 生产工单未发布，MES 不允许开工
- 工单未按版本生效，现场不允许领用新 BOM
- MES 未确认完工，WMS 不允许成品入库
- WMS 未入库，ERP 不允许完成库存记账
- QMS 未裁决合格，受控批次不得转入可用库存
- ECN 未完成审批发布，不得影响执行侧版本切换

这些规则后续可以做成：

- 页面状态限制
- 按钮禁用条件
- mock 校验规则
- 接口前置校验

## 10. 对静态页开发顺序的直接影响

当前系统采用“静态页先于 mock”的策略，因此页面优先级必须服从数据链优先级。

建议后续静态页顺序调整为：

1. `mdm/物料`
2. `mdm/单位、分类、编码规则、仓库与库位`
3. `plm/bom`
4. `plm/ecn`
5. `plm/工艺路线`
6. `mes/工单`
7. `mes/报工与在制`
8. `wms/收货、入库、发料`
9. `qms/来料检验、过程检验`
10. `erp/库存台账、订单状态`

不建议当前优先做重分析、轻对象的页面，例如：

- 复杂 BI 看板
- 纯经营分析页
- 脱离业务主链的孤立统计页

## 11. 当前阶段不建议过早重投入的内容

以下方向是对的，但不应成为当前阶段主任务：

- ESB / iPaaS 平台建设
- 全量实时事件总线
- 全系统秒级同步
- 全量数据仓治理
- 一开始就做 AI 驱动决策

当前更合理的顺序是：

1. 统一对象
2. 统一口径
3. 统一状态流
4. 再进入 mock
5. 再进入 API
6. 最后做深度集成优化

## 12. 第一阶段交付物建议

为了让这份蓝图真正落地，建议下一阶段补齐以下正式产物：

### 12.1 主数据对象清单

至少包含：

- 对象名称
- 对象定义
- 所属层级
- 主责模块
- 创建来源
- 审批方式
- 发布方式
- 消费模块

### 12.2 核心字段口径表

优先覆盖：

- 物料
- BOM
- 工艺路线
- 仓库/库位
- 工单
- 库存状态

### 12.3 跨模块状态流说明

优先覆盖：

- 销售订单状态
- 生产工单状态
- 报工状态
- 入库状态
- 检验状态
- ECN 生效状态

### 12.4 三条主链的静态页蓝图

要求：

- 页面字段围绕统一对象
- 页面动作符合状态流
- 页面之间存在清晰跳转关系

## 13. 最终结论

对玄工智链来说，当前最关键的不是“把所有模块先铺满”，而是先把以下三件事做对：

- 把 `MDM` 升级为统一主数据中心
- 把 `PLM / MES / ERP / WMS / QMS` 之间的核心对象关系定义清楚
- 把跨模块业务规则提前固化到静态页与后续 mock 设计中

如果这一步做对，后面的静态页、mock、接口、集成都会自然顺下来。  
如果这一步没做对，后面开发越快，返工成本越高。
