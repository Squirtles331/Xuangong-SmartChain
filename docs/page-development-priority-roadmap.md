# 玄工智链页面开发优先级路线图

## 1. 文档定位

本文档用于把玄工智链当前的一体化设计思路，转换为可执行的静态页面开发顺序。其目标不是讨论最终接口实现，而是明确：

- 静态页阶段先做哪些页面
- 哪些页面必须成组推进
- 哪些页面依赖上游对象先稳定
- mock 与 API 分别在什么阶段介入

本文档默认遵循以下原则：

- 先静态页面与静态数据，后 mock，最后 API
- 先主数据与基础对象，后业务单据与执行闭环
- 先单模块自洽，后跨模块联动
- 先“页面结构正确”，后“业务动作联通”

## 2. 当前阶段的开发原则

### 2.1 页面开发目标不是“把页面铺满”，而是“先建立一体化骨架”

当前系统最需要解决的不是单个 CRUD 页数量，而是以下三个问题：

- 页面归属要清楚，不能继续出现对象跨模块混放
- 对象口径要统一，不能同一对象在多个页面各自表达
- 状态流要前置，不能等到 mock 或接口阶段再补业务约束

因此，页面优先级不应以“哪个模块功能多”决定，而应以“哪个页面最能稳定全局对象关系”决定。

### 2.2 优先级判断标准

一个页面是否应该优先开发，按照以下标准排序：

1. 是否承载核心主数据对象
2. 是否会被多个模块复用
3. 是否决定后续页面命名、字段、状态、操作规范
4. 是否能在没有真实接口的前提下先完成静态表达
5. 是否是后续 mock 和 API 的基础入口页

## 3. 分阶段路线图总览

| 阶段     | 目标定位               | 重点模块                | 页面类型                           | 是否允许 mock  |
| -------- | ---------------------- | ----------------------- | ---------------------------------- | -------------- |
| 第一阶段 | 稳定统一对象骨架       | `MDM` `PLM`             | 主数据页、定义页、版本页           | 否，先静态数据 |
| 第二阶段 | 建立制造执行主链路     | `MES` `WMS` `QMS` `APS` | 工单页、执行页、报工页、库存事务页 | 局部允许       |
| 第三阶段 | 打通经营协同链路       | `ERP` `SRM` `CRM`       | 订单页、采购页、供应协同页、客户页 | 允许           |
| 第四阶段 | 建立设备连接与分析展示 | `EAM` `IOT` `BI`        | 设备页、监控页、分析页、看板页     | 允许           |
| 第五阶段 | 收口平台能力与系统治理 | `SYS` 全局              | 权限、字典、参数、审计、集成配置   | 允许           |

## 4. 第一阶段：核心对象静态页优先

### 4.1 阶段目标

先把整个系统最基础、最容易产生口径分裂的对象定义清楚，并把页面结构、字段分组、状态表达、操作按钮全部稳定下来。

### 4.2 本阶段必须完成的页面组

#### A. MDM 主数据基础页

- 物料主数据
- 物料分类
- 计量单位
- 仓库
- 库区库位
- 供应商主数据
- 客户主数据
- 组织/部门/班组

#### B. PLM 产品与工艺定义页

- 产品主数据
- EBOM
- MBOM
- 工艺路线
- 工序定义
- 工程变更单（ECN）

### 4.3 本阶段的页面产出标准

每个页面至少完成以下内容：

- 页面标题、面包屑、模块归属准确
- 查询区、工具栏、表格列、详情区结构稳定
- 表单字段分组完成
- 状态标签口径统一
- 操作按钮可见性按状态区分
- 详情页或抽屉中的“上下游关系区”预留完成

### 4.4 本阶段暂不追求的内容

- 不做真实接口联调
- 不做跨系统实时同步
- 不做复杂审批流引擎
- 不做导入导出真实逻辑

### 4.5 本阶段完成标志

- `MDM` 与 `PLM` 的核心对象页面可以独立演示
- 页面中的字段命名、状态值、操作规则可作为后续模块统一基线
- 后续页面不再自行新增核心对象字段口径

## 5. 第二阶段：制造执行与库存质量主链路

### 5.1 阶段目标

在第一阶段对象定义稳定后，开始建设真正体现“计划-生产-报工-入库-检验”主链路的静态页面。

### 5.2 本阶段优先页面组

#### A. APS 计划协同页

- 主生产计划
- 生产排程
- 工单排产视图

#### B. MES 执行页

- 生产工单
- 生产派工
- 工序执行
- 生产报工
- 不良登记

#### C. WMS 库存事务页

- 入库单
- 出库单
- 领料单
- 退料单
- 库存台账

#### D. QMS 质量页

- 来料检验
- 过程检验
- 完工检验
- 不合格评审

### 5.3 本阶段页面设计重点

- 让页面能体现对象前后依赖关系，而不是只保留单表 CRUD
- 在详情页中展示来源单据、下游去向、状态轨迹
- 对关键单据建立“头-行-执行记录”结构
- 对车间场景优先使用更明确的状态块、步骤块、执行卡片

### 5.4 本阶段 mock 介入规则

本阶段可以开始引入第一批 mock，但只用于以下场景：

- 单据列表与详情静态切换
- 状态流转演示
- 单据头行结构联动
- 明细子表展示

本阶段不建议过早进入以下 mock：

- 多系统自动回写
- 复杂库存即时演算
- 真实排程算法结果
- 设备实时采集

### 5.5 本阶段完成标志

- 从 BOM/工艺到工单，再到报工/入库/检验的页面链路能完整演示
- 用户能在页面层面理解业务主线，而不依赖后端逻辑解释
- mock 数据结构可作为后续接口 DTO 草稿

## 6. 第三阶段：经营协同与供应链页面

### 6.1 阶段目标

把生产执行主链条向外延伸到经营侧与协同侧，建立订单、采购、供应、客户的业务闭环。

### 6.2 本阶段优先页面组

#### A. ERP 经营页

- 销售订单
- 采购申请
- 采购订单
- 委外订单
- 应收应付台账

#### B. SRM 协同页

- 供应商门户首页
- 询报价
- 送货协同
- 对账协同

#### C. CRM 客户页

- 客户档案
- 客户机会
- 客户订单跟踪
- 售后服务请求

### 6.3 本阶段页面设计重点

- 订单类页面优先体现来源、执行进度、交付状态
- 采购与供应协同页面优先体现对账、送货、来料关联
- 客户侧页面优先体现订单视角，不再单独堆砌信息管理页

### 6.4 本阶段完成标志

- 经营侧页面与制造侧页面在对象命名和状态定义上保持一致
- 用户可从订单视角追踪到生产、库存、质量的关键节点

## 7. 第四阶段：设备、采集与分析展示

### 7.1 阶段目标

在核心业务主链稳定后，再建设对实时性与展示性要求更高的设备和分析能力，避免早期把精力耗在“看起来很高级”的界面上。

### 7.2 本阶段优先页面组

#### A. EAM 设备管理页

- 设备台账
- 点检计划
- 保养计划
- 维修工单

#### B. IOT 连接采集页

- 设备连接管理
- 点位配置
- 实时监控
- 报警记录

#### C. BI 分析页

- 生产分析看板
- 质量分析看板
- 库存分析看板
- 经营分析看板

### 7.3 本阶段页面设计重点

- 分析页面必须建立在前面对象定义一致的基础上
- 看板不应先于业务对象落地
- 实时监控页先做结构和展示布局，不提前承诺真实秒级数据能力

## 8. 第五阶段：平台治理与系统能力收口

### 8.1 阶段目标

把系统参数、权限、字典、消息、审计、集成配置等平台能力作为最后收口阶段统一治理，而不是零散穿插在业务开发过程中。

### 8.2 本阶段优先页面组

- 用户与角色
- 菜单与权限
- 数据字典
- 编码规则
- 系统参数
- 消息模板
- 接口日志
- 数据同步任务
- 审计日志

### 8.3 本阶段页面设计重点

- 平台页命名要服务管理员角色，不与业务页混淆
- 配置项必须与前面对象体系一一对应
- 系统设置页面优先解决“可治理性”，而不是堆砌字段

## 9. 页面开发批次建议

为了避免页面过多同时推进导致标准再次失控，建议采用以下批次节奏：

### 批次一：对象定义批

- 物料
- BOM
- 工艺路线
- 仓库/库位
- 客户/供应商

目标是先定对象、字段、状态、基础交互。

### 批次二：执行主链批

- 生产工单
- 生产报工
- 领料/退料
- 入库单
- 检验单

目标是先定单据结构、流转关系、明细区布局。

### 批次三：经营协同批

- 销售订单
- 采购订单
- 委外单
- 供应商协同
- 客户跟踪

目标是把经营侧与制造侧挂接起来。

### 批次四：分析治理批

- 看板
- 监控
- 系统设置
- 日志审计

目标是建立展示层与治理层收口。

## 10. 页面依赖关系建议

| 页面类型       | 前置依赖             | 原因                         |
| -------------- | -------------------- | ---------------------------- |
| 工单页         | 物料、BOM、工艺路线  | 工单不能脱离产品定义独立存在 |
| 报工页         | 工单、工序定义       | 报工是对执行过程的反馈       |
| 入库页         | 报工、完工结果       | 入库不应先于完工事实         |
| 检验页         | 来料/工序/完工对象   | 质量对象必须挂在业务对象上   |
| 销售订单跟踪页 | 工单、库存、发货信息 | 订单页需要跨模块结果汇总     |
| 分析看板页     | 前述核心对象全部稳定 | 看板不能先于底层对象口径     |

## 11. 静态页、mock、API 的交接规则

### 11.1 静态页阶段

要求完成：

- 页面布局
- 字段清单
- 列表列定义
- 状态标签
- 按钮可见性
- 详情区块划分

输出物包括：

- 页面静态数据文件
- 字段说明
- 列表列配置
- 页面操作说明

### 11.2 mock 阶段

要求在静态页稳定后才进入，重点补充：

- 列表查询条件联动
- 详情数据结构
- 状态流转返回
- 行项目与子记录结构

输出物包括：

- mock 响应结构
- 页面数据映射表
- 状态流转样例

### 11.3 API 阶段

要求在 mock 稳定后才进入，重点补充：

- DTO 定义
- 字段校验规则
- 错误码与异常提示
- 跨模块接口边界

输出物包括：

- 接口清单
- 请求响应模型
- 字段约束说明
- 集成说明

## 12. 当前最推荐的落地顺序

如果从现在开始正式推进，建议按以下顺序执行：

1. 先完成 `MDM` 的物料、仓库、客户、供应商四类主数据页
2. 再完成 `PLM` 的 BOM、工艺路线、ECN 三类定义页
3. 接着完成 `MES` 的工单、报工两个核心页
4. 再补 `WMS` 的领料、入库、库存台账页
5. 然后补 `QMS` 的来料检验、过程检验、完工检验页
6. 最后再进入 `ERP` `SRM` `CRM` `BI` 等经营协同与分析层

这条路线的核心价值在于：

- 先把“对象真相”建立起来
- 再把“执行真相”建立起来
- 最后再做“经营协同”和“分析展示”

这样做，后续无论是 mock、API 还是最终集成都不会反复返工。
