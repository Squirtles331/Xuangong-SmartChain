# 玄工智链菜单模块开发批次清单

## 1. 文档目的

本文档用于把当前玄工智链的页面开发顺序，从“对象优先级”和“页面优先级”进一步收口为“菜单模块批次顺序”。

它重点回答三个问题：

- 主数据完成后，下一步先开发哪些菜单模块
- 各菜单模块内部应先做哪些页面，再做哪些页面
- 哪些模块现在不建议提前进入开发

本文档默认遵循以下前提：

- 先静态页面与静态数据，后 mock，后 API
- 先上游定义，后执行链路，后经营协同，最后分析治理
- 先建立制造一体化主链，再扩展外围模块

- 每一批推进到什么程度再进入下一批

## 3. 当前前提判断

结合当前系统状态，可作为后续开发起点的共识如下：

1. `MDM` 主数据基础页已基本完成，已能作为后续模块的统一对象来源
2. 当前阶段不进入 `mock` 设计，继续推进静态页与静态数据
3. 菜单模块不再平均铺开，而是围绕制造主链做分批推进

因此，后续模块顺序不建议按“系统缩写并列展开”，而建议按业务主链展开：

```text
PLM -> MES -> WMS -> QMS -> APS -> ERP -> SRM -> CRM -> EAM -> IOT -> BI -> SYS
```

## 4. 总体开发顺序

| 批次 | 主模块 | 目标定位 | 是否建议现在开始 |
| --- | --- | --- | --- |
| 第一批 | `PLM` | 稳定产品定义与工艺定义 | 是 |
| 第二批 | `MES` | 拉起制造执行主链入口 | 是 |
| 第三批 | `WMS` | 承接领料、入库、库存事务 | 是 |
| 第四批 | `QMS` | 把质量检验挂接到执行链 | 是 |
| 第五批 | `APS` `ERP` | 建立计划与经营单据链 | 是，但晚于执行链 |
| 第六批 | `SRM` `CRM` | 建立外部协同与客户闭环 | 是，但晚于 ERP |
| 第七批 | `EAM` `IOT` | 建立设备与采集支撑 | 否，后置 |
| 第八批 | `BI` `SYS` | 建立分析与治理收口 | 否，最后收口 |

## 5. 第一批：PLM 定义链

### 5.1 为什么先做

虽然 `MDM` 已经完成基础主数据，但制造一体化主链还缺少真正驱动下游的产品定义与工艺定义。

如果 `PLM` 不先稳定，后面的工单、领料、报工、检验都会出现以下问题：

- BOM 口径漂移
- 工艺路线表达不统一
- 版本变更无法前置体现
- 页面字段与状态反复返工

### 5.2 本批建议菜单顺序

1. `BOM列表`
2. `BOM编辑`
3. `工程变更`
4. `工艺路线`
5. `工序定义`
6. `BOM对比`
7. `BOM展开`

### 5.3 本批完成标准

- `EBOM / MBOM` 页面表达稳定
- 工艺路线与工序定义结构稳定
- `ECN` 页面已能表达版本变更、影响对象、状态流
- 后续 `MES / WMS / QMS` 不再自行创造 BOM、工艺字段口径

## 6. 第二批：MES 执行链

### 6.1 为什么第二批做

`MES` 是制造执行主链的核心模块，必须建立在 `PLM` 已稳定的前提下。

这一批不是为了先做复杂逻辑，而是为了把执行真相页面结构定下来：

- 工单头
- 工单行
- 工序执行块
- 报工记录
- 任务与追溯关系

### 6.2 本批建议菜单顺序

1. `工单列表`
2. `工单详情`
3. `新建工单`
4. `报工中心`
5. `我的任务`
6. `工单追溯`
7. `工单看板`

### 6.3 本批完成标准

- 工单列表、详情、执行关系清楚
- 报工页面已能表达工序执行和状态变化
- `MES` 页面开始形成“头表 + 行表 + 执行记录 + 状态轨迹”统一结构

## 7. 第三批：WMS 仓储事务链

### 7.1 为什么紧跟 MES

生产执行一旦落地，下一步自然就是物料流和库存流。

`WMS` 不应该孤立做，而应明确承接：

- 生产领料
- 退料
- 完工入库
- 库存台账

### 7.2 本批建议菜单顺序

1. `收货入库`
2. `拣料出库`
3. `库存台账`
4. `退料`
5. `盘点管理`
6. `库内调拨`

### 7.3 本批完成标准

- 仓储事务页能表达来源单据和事务方向
- 库存台账已作为库存真相入口页存在
- `MES -> WMS` 页面链路能完整演示领料与入库关系

## 8. 第四批：QMS 质量链

### 8.1 为什么不要提前单做

质量页如果脱离业务对象单独开发，很容易退化成普通 CRUD。

更合理的方式是把 `QMS` 明确挂在执行链上，让检验对象有真实来源：

- 来料检验挂采购/收货
- 过程检验挂工序执行
- 完工检验挂完工入库

### 8.2 本批建议菜单顺序

1. `检验模板`
2. `来料检验`
3. `过程检验`
4. `完工检验`
5. `不合格评审`

### 8.3 本批完成标准

- 三类检验页面已明确拆分
- 检验模板、检验任务、裁决区块关系清楚
- `QMS` 已从“泛质量页面”转为“挂靠业务对象的质量链页面”

## 9. 第五批：APS 与 ERP 计划经营链

### 9.1 为什么放在执行链后面

`APS` 和 `ERP` 很重要，但不适合先于制造执行主链开发。

原因是：

- 没有稳定的工单与库存对象，计划页容易空心化
- 没有执行和质量真相，经营页容易只剩表面单据

### 9.2 本批建议模块顺序

先 `APS`，后 `ERP`。

### 9.3 APS 建议菜单顺序

1. `主生产计划`
2. `生产排程`
3. `工单排产视图`

### 9.4 ERP 建议菜单顺序

1. `MRP结果`
2. `销售订单`
3. `采购申请`
4. `采购订单`
5. `委外订单`
6. `财务台账`

### 9.5 本批完成标准

- 计划页能引用上游物料、BOM、工艺、库存信息
- 订单页能关联工单、库存、交付、检验关键节点
- `ERP` 页面不再只是孤立经营页，而开始与制造链挂接

## 10. 第六批：SRM 与 CRM 协同链

### 10.1 为什么晚于 ERP

外部协同页必须依附内部对象真相，否则只会形成新的信息孤岛。

所以更合理的顺序是：

- 先把订单、采购、库存、工单这些内部对象稳定
- 再把供应协同与客户协同接进来

### 10.2 SRM 建议菜单顺序

1. `询报价`
2. `送货协同`
3. `对账协同`

### 10.3 CRM 建议菜单顺序

1. `客户机会`
2. `客户订单跟踪`
3. `售后服务请求`

### 10.4 本批完成标准

- `SRM` 页面能挂接采购、来料、对账对象
- `CRM` 页面能挂接订单、交付、售后对象
- 外协与客户协同不再停留在信息登记层

## 11. 第七批：EAM 与 IOT 设备链

### 11.1 当前为什么后置

这类页面看起来很有“智能制造感”，但如果太早做，容易消耗大量精力在展示和实时感上，而不是业务主链本身。

因此当前建议后置。

### 11.2 EAM 建议菜单顺序

1. `设备台账`
2. `点检计划`
3. `保养计划`
4. `维修工单`

### 11.3 IOT 建议菜单顺序

1. `设备连接管理`
2. `点位配置`
3. `实时监控`
4. `报警记录`

## 12. 第八批：BI 与 SYS 收口链

### 12.1 BI 为什么最后做

看板和分析页面必须建立在前面对象和状态口径一致的基础上。

否则看板做得越早，返工越大。

### 12.2 BI 建议菜单顺序

1. `生产分析看板`
2. `质量分析看板`
3. `库存分析看板`
4. `经营分析看板`

### 12.3 SYS 为什么最后收口

系统治理能力很重要，但当前阶段应以业务主链为先，治理类页面放在最后统一收口更合理。

### 12.4 SYS 建议菜单顺序

1. `用户与角色`
2. `菜单与权限`
3. `数据字典`
4. `编码规则`
5. `系统参数`
6. `消息模板`
7. `接口日志`
8. `数据同步任务`
9. `审计日志`

## 13. 当前最推荐的正式推进节奏

如果从现在开始继续推进静态页开发，建议按以下节奏执行：

### 第一轮

- `PLM`
- `MES`

目标：

- 把产品定义和生产执行主链先拉起来

### 第二轮

- `WMS`
- `QMS`

目标：

- 把库存事务和质量裁决挂接到执行链

### 第三轮

- `APS`
- `ERP`

目标：

- 把计划和经营单据建立在制造主链之上

### 第四轮

- `SRM`
- `CRM`

目标：

- 把外部协同与客户协同接入闭环

### 第五轮

- `EAM`
- `IOT`
- `BI`
- `SYS`

目标：

- 完成设备支撑、分析展示和平台治理收口

## 14. 当前不建议的推进方式

当前不建议继续采用以下做法：

1. 所有模块同时平均铺开
2. 先做看板和监控，再补业务对象
3. 先做经营单据，再反推制造执行
4. 质量页脱离执行链单独设计
5. 平台治理页穿插打断主业务链开发

## 15. 结论

在 `MDM` 已完成的前提下，后续最合理的菜单模块开发顺序应固定为：

```text
PLM -> MES -> WMS -> QMS -> APS -> ERP -> SRM -> CRM -> EAM -> IOT -> BI -> SYS
```

如果只保留一句执行口令，可以直接理解为：

```text
先定义
后执行
再库存与质量
再计划与经营
最后协同、设备、分析、治理
```
