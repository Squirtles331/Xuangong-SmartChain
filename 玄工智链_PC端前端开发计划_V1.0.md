# 玄工智链\_PC端前端开发计划\_V1.0

## 一、计划目标

本计划用于指导玄工智链 PC 端前端重构后的后续建设顺序。

当前阶段目标不是一边做 mock 一边补页面，而是先把以下三件事稳定下来：

1. 菜单和页面归属稳定
2. 静态页面结构稳定
3. 静态数据字段稳定

在这三件事稳定之前，不进入大规模 mock 集成。

## 二、当前建设总原则

### 1. 架构先行

所有页面设计必须服从当前正式基线：

- 一级菜单已拆分为 `工作台 + 13 个独立业务域`
- 页面显示名统一使用中文产品语言
- 页面归属以业务域拆分结果为准

### 2. 静态优先

当前统一执行：

```text
路由与菜单基线
-> 页面静态结构
-> 静态数据字段
-> 页面内跳转
-> mock 集成
-> API 联调
```

### 3. 页面服务对象，不服务旧模块名

优先围绕以下对象建设页面：

- 工单
- 工序任务
- WIP
- 检验任务
- 不合格处置
- 返工链
- BOM / 工艺路线
- 排程结果
- 计划需求
- 库存批次
- 设备事件

### 4. 工作台先做行动入口，不做经营大屏

首页必须优先承接：

- 待办
- 预警
- 执行摘要
- 协同域摘要
- 快捷入口

不再把首页当成经营大屏先做。

## 三、静态页阶段交接规则

### 1. 页面进入 mock 前必须满足

1. 菜单名称、页面标题、面包屑名称已与最终中文口径一致
2. 页面已归属到正确一级业务域
3. 页面主责系统、协同系统、核心对象已确认
4. 页面静态数据字段结构已稳定
5. 页面默认跳转关系已确定
6. 页面空态、占位态、状态标签文案已稳定

### 2. 页面未满足以上条件时，禁止

1. 先做接口字段映射
2. 先造 mock 接口协议
3. 先写复杂联动逻辑
4. 先写与页面定位冲突的按钮和操作流

### 3. 统一 handoff 规则

所有静态页交接给 mock 或接口阶段前，都必须先经过一轮“菜单真相表核对”：

- 页面名是否正确
- 页面放置位置是否正确
- 页面责任归属是否正确
- 页面是否表达了正确的核心对象

如果任一项不通过，该页面不得进入 mock 阶段。

## 四、推荐建设顺序

## 4.1 第一阶段：基线收口

目标：把所有后续工作建立在统一导航与统一命名上。

本阶段任务：

- 对齐一级业务域命名
- 对齐关键二级、三级页面命名
- 对齐页面归属和边界说明
- 固定工作台蓝图

交付结果：

- 菜单架构文档可作为正式依据
- 工作台蓝图可直接指导静态实现

## 4.2 第二阶段：工作台静态化

目标：先把用户登录后的统一入口稳定下来。

页面范围：

- 工作台首页

模块顺序固定为：

1. 我的待办
2. 今日异常与预警
3. 执行总览
4. 协同域摘要
5. 快捷入口
6. 经营摘要

本阶段只做：

- 静态布局
- 静态卡片
- 静态数据 provider
- 跳转占位

本阶段不做：

- 实时数据刷新
- 复杂角色化首页
- 大型经营图表墙

## 4.3 第三阶段：执行主链静态页

目标：优先把最关键的执行闭环页面做完整。

推荐顺序：

1. `生产执行 / 工单`
2. `生产执行 / 工序执行`
3. `生产执行 / 追溯与异常`
4. `质量管理 / 检验任务、不合格处置、返工裁决`
5. `仓储物流 / 库存台账、生产领料、批次隔离`

原因：

- 这一组页面共同构成“工单 -> 工序 -> WIP -> 检验 -> 异常 -> 返工/放行”的主闭环

## 4.4 第四阶段：工程计划静态页

目标：把执行前置的工程与计划页面补完整。

推荐顺序：

1. `产品工艺 / 产品结构清单`
2. `产品工艺 / 工艺路线`
3. `产品工艺 / 工程变更`
4. `产品工艺 / 版本切换`
5. `计划排程 / 排程结果、排程约束、排程历史`
6. `经营计划 / 需求池、MRP、预测需求`

原因：

- 执行页稳定后，再补执行前置约束，返工更少

## 4.5 第五阶段：协同与经营静态页

目标：把外围协同和管理分析域补齐。

推荐顺序：

1. `采购供应`
2. `设备管理`
3. `设备物联`
4. `客户经营`
5. `经营分析`
6. `系统平台`
7. `主数据`

原因：

- 这些页面对平台完整性重要，但不是最早阻塞静态主链的部分

## 五、每个阶段的统一产出

每个页面在静态阶段都至少要产出：

1. 页面标题与归属
2. 页面模块结构
3. 静态数据字段定义
4. 查询区和列表区结构
5. 主要状态标签
6. 主操作按钮
7. 默认跳转关系

## 六、静态数据建设规则

### 1. 先定字段，再定来源

静态阶段先确定字段结构，不用先纠结接口来源。

### 2. 同一页面优先单独 provider

建议每个页面先使用本地静态 provider，保证：

- 字段独立
- 可快速替换
- 不被 mock 耦合

### 3. 聚合页使用统一聚合对象

像工作台这类聚合页，优先使用统一对象承接，而不是边写页面边拼散数据。

## 七、mock 接入顺序

只有静态页稳定后，再按下面顺序接 mock：

1. 工作台
2. 生产执行
3. 质量管理
4. 仓储物流
5. 产品工艺
6. 计划排程
7. 经营计划
8. 采购供应
9. 设备管理
10. 设备物联
11. 客户经营
12. 经营分析
13. 系统平台
14. 主数据

接 mock 时的原则：

- 保持静态阶段字段结构不变
- mock 适配页面，不让页面反向迁就 mock
- 先接列表和摘要，再接复杂动作流

## 九、页面评审口径

一个页面评审时，不只看“能不能打开”，还要看：

1. 名称是否像产品，不像技术模块
2. 放的位置是否对
3. 责任归属是否对
4. 页面是否围绕核心对象展开
5. 页面动作是否符合主责边界
6. 后续是否容易接 mock 和接口

## 十、结论

当前前端重构后的后续推进方式，必须统一为：

```text
先菜单
先静态页
先静态数据
后 mock
后接口
```

只要坚持这个顺序，后续页面建设就会更稳，也能避免一边做 mock 一边返工菜单和归属的问题再次出现。
