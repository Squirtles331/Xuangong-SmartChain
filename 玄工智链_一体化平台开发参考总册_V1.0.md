# 玄工智链\_一体化平台开发参考总册\_V1.0

## 一、文档定位

本文档用于把当前最影响开发落地的架构、菜单、对象主责和实施规则，收敛成一份统一参考。

它主要服务于：

- 产品与页面设计
- 前后端开发
- 静态页建设
- mock 与接口联调
- 测试验收

本文档默认建立在当前已重构完成的路由基线之上，不再沿用旧的混合菜单分组口径。

## 二、平台一句话定义

玄工智链是一套：

> 以 `MES` 为执行中枢，以对象、事件、状态和审批放行为核心机制，并由 `PLM / APS / WMS / QMS / SRM / EAM / IoT / CRM / ERP / BI / SYS / MDM` 协同支撑的一体化制造平台。

## 三、开发必须记住的十条基线

1. `MES` 是执行中枢，但不是所有对象的唯一拥有者。
2. 当前前端菜单已经拆成独立业务域，后续页面不得再回退到旧的混合域命名。
3. 当前最小控制粒度锁定为：`工单 + 工序任务 + WIP 批次`。
4. `领料不等于消耗`，`WMS` 管库存移动真相，`MES` 管现场消耗真相。
5. `QMS` 负责质量裁决，不只是登记检验结果。
6. `返工裁决` 与 `返工执行` 必须拆开。
7. `版本发布` 不等于 `版本切换完成`。
8. 异常必须遵循：`先锁定 -> 再处置 -> 再放行`。
9. 工作台只做聚合和导流，不维护业务对象真相。
10. 静态页必须先对齐菜单真相表，再进入 mock 集成。

## 五、开发优先围绕的核心对象

当前版本优先围绕以下对象建设：

1. 销售订单
2. 计划需求
3. APS 排程结果
4. 生产工单
5. 工序任务
6. WIP 批次
7. 投料 / 消耗记录
8. 检验任务
9. 不合格处理单
10. BOM / 工艺路线版本
11. 版本切换任务
12. 设备事件

其中第一优先链路为：

```text
工单 -> 工序任务 -> WIP -> 检验结论 -> 异常锁定/恢复 -> 返工或放行
```

## 七、关键边界裁决

### 7.1 领料与消耗

- `仓储物流 / 生产领料`：表达库存移动
- `生产执行 / 投料与消耗`：表达现场实际消耗

### 7.2 返工

- `质量管理 / 返工裁决`：决定是否返工
- `生产执行 / 返工执行`：承接返工链执行

### 7.3 版本切换

- `产品工艺 / 工程变更`：定义版本发布
- `产品工艺 / 版本切换`：定义现场切换任务

### 7.4 异常处置

- 页面统一名称使用 `异常处置`
- 固定语义为：`锁定 -> 处置 -> 放行`
- 质量结论不得绕过 `QMS`

### 7.5 工作台与经营分析

- 工作台回答：`现在要做什么`
- 经营分析回答：`最近经营表现怎么样`

## 八、静态页阶段统一规则

### 8.1 静态优先原则

当前阶段严格执行：

```text
菜单基线 -> 静态页面 -> 静态数据 -> 页面跳转 -> mock -> API
```

### 8.2 静态页交接门槛

一个页面进入 mock 之前，至少必须满足以下五项：

1. 菜单名称与面包屑名称已对齐最终中文口径
2. 页面所属业务域已与当前路由拆分方式一致
3. 页面主责系统、协同系统、核心对象已确认
4. 静态数据字段结构已稳定
5. 页面默认跳转关系已按工作台蓝图或菜单真相表确定

### 8.3 当前不允许的做法

1. 页面还没定名，就先做 mock 接口
2. 页面还没定归属，就先把按钮和数据流写死
3. 用 mock 返回结构反向决定页面字段
4. 用旧菜单术语继续扩页面

## 十、页面设计检查清单

每新增一个页面，先回答：

1. 它服务哪个核心对象
2. 它属于哪个一级业务域
3. 它的主责系统是谁
4. 它触发哪个状态动作
5. 它是否涉及审批放行
6. 它是否涉及跨系统联动
7. 它是否已经具备静态阶段可用的数据字段

如果这些问题答不出来，这个页面大概率仍是“功能点页面”，不是平台型页面。

## 十三、结论

当前开发参考口径已经切换为：

- 独立业务域导航
- 中文产品命名
- 对象主责清晰
- 静态优先、mock 后置

后续继续开发时，先看菜单域、对象主责和静态页交接门槛，再决定是否进入 mock 和接口阶段，这样才能避免再次把错误的信息架构固化进系统。
