# 玄工智链_一体化平台开发参考总册_V1.0

## 一、文档定位

本文档用于把当前玄工智链一体化平台架构设计中，最影响开发落地的内容收拢为一份统一参考。

目标是让以下角色优先看这一份文档即可进入工作状态：

- 产品设计
- 前后端开发
- 接口联调
- 审批流配置
- 测试验收
- 项目实施

本文档不是替代全部原始文档，而是作为开发阶段的统一入口。

---

## 二、平台一句话定义

玄工智链是一套：

> 以 `MES` 为执行中枢，以事件驱动、状态控制、审批放行为核心机制，以外围专业系统协同闭环的一体化离散制造平台。

---

## 三、开发必须先记住的九条平台基线

1. `MES` 是执行中枢，但不是所有主数据和交易数据的唯一拥有者。
2. 当前最小控制粒度锁定为：`工单 + 工序任务 + WIP 批次`。
3. `返工` 必须走独立返工路径，不能直接拖回主流程。
4. `领料不等于消耗`，`WMS` 管库存真相，`MES` 管生产消耗真相。
5. `QMS` 负责质量裁决，不只是记录检验结果。
6. `版本发布 != 版本切换完成`，切换必须受控。
7. 异常必须先锁定，再处置，再放行。
8. 扰动分层处理：`L1 现场微调 / L2 局部重排 / L3 全局重排`。
9. 平台统一运行机制是：`事件驱动 + 状态控制 + 审批放行`。

后续任何页面、接口、状态流、审批流设计，如果与这九条冲突，以这九条为准。

## 五、开发优先围绕的核心对象

当前版本必须优先围绕这些对象建设，不建议脱离对象只做孤立功能页：

1. 销售订单
2. 计划需求
3. APS 排程结果
4. 生产工单
5. 工序任务
6. WIP 批次
7. 投料 / 消耗记录
8. 检验任务
9. 不合格处理单
10. BOM / 工艺路线版本
11. 版本切换任务
12. 设备事件

其中开发优先级最高的是：

```text
工单 -> 工序任务 -> WIP 批次 -> 检验结论 -> 异常锁定 / 恢复 -> 版本切换
```

## 六、核心对象主责矩阵

## 6.1 统一原则

- 一个对象只能有一个主责系统
- 执行对象优先归 `MES`
- 库存对象优先归 `WMS`
- 质量裁决对象优先归 `QMS`
- 工程定义对象优先归 `PLM`
- 计划结果由 `ERP / APS` 主责，`MES` 承接执行

## 6.3 开发禁止性规则

以下动作默认禁止：

1. `ERP / CRM / APS` 直接改 `MES` 执行状态。
2. `MES` 直接改 `WMS` 库存真相。
3. `MES` 绕过 `QMS` 直接关闭质量异常。
4. `WMS` 用领料结果替代 `MES` 消耗事实。
5. `PLM` 发布版本后直接视为现场已切换完成。

## 七、核心事件清单

## 7.1 事件设计原则

- 事件表达“事实已发生”
- 一个事件只表达一个核心事实
- 事件必须可追溯到业务对象
- 下游消费必须幂等

## 7.2 标准事件头建议

每个事件至少带：

- `event_id`
- `event_type`
- `source_system`
- `object_type`
- `object_id`
- `occurred_at`
- `published_at`
- `status_before`
- `status_after`
- `operator_id`
- `trace_id`
- `version`

## 八、核心状态机定义

## 8.1 生产工单状态

### 状态集合

`draft -> pending_release -> waiting_kitting -> ready -> in_progress -> waiting_inspection -> completed -> closed`

异常分支：

`in_progress -> paused`

`in_progress / paused / waiting_inspection -> locked`

恢复分支：

`locked -> in_progress`

### 关键规则

- `waiting_kitting` 之前不能开工
- `locked` 状态不得继续执行
- 强制关闭必须审批

## 8.2 工序任务状态

### 状态集合

`created -> waiting_precondition -> ready -> running -> reported -> waiting_quality -> done -> closed`

异常分支：

`running -> paused`

`running / reported / waiting_quality -> locked`

恢复分支：

`locked -> ready / running`

### 关键规则

- 工序任务是现场最直接的执行状态对象
- 自动报工、异常、IoT 采集都应先落到工序任务

## 8.3 WIP 批次状态

### 状态集合

`created -> waiting_process -> processing -> waiting_transfer -> waiting_inspection -> released -> completed -> closed`

异常分支：

`processing / waiting_transfer / waiting_inspection -> frozen`

返工分支：

`frozen -> rework_pending -> reworking -> waiting_inspection`

### 关键规则

- `WIP` 不是库存批次
- `WIP` 由 `MES` 主责，不由 `WMS` 主责

## 8.4 检验任务状态

`created -> assigned -> inspecting -> pending_review -> passed / failed -> closed`

## 8.5 不合格处理单状态

`created -> under_review -> rework_decided / concession_decided / scrap_decided -> executing -> reinspection_pending -> closed`

## 8.6 版本切换任务状态

`requested -> impact_analyzing -> pending_approval -> approved -> executing -> partially_completed / completed -> closed`

### 关键规则

- 只要旧版未收口完成，就不能直接记为 `completed`

## 九、审批放行点

当前开发必须预留审批的关键动作：

1. 特采放行
2. 报废确认
3. 异常解锁
4. 强制关闭工单
5. 替代料启用
6. 版本切换放行

原则：

- 审批不是留痕，而是状态迁移许可
- 审批通过后必须触发明确状态变化

## 十、跨系统集成规则

## 10.1 关键集成矩阵

| 来源系统 | 目标系统 | 核心对象 | 集成目标 | 推荐方式 |
| --- | --- | --- | --- | --- |
| `CRM` | `ERP / APS / MES` | 销售订单、订单变更 | 形成计划需求和扰动来源 | 事件 + 查询 |
| `ERP` | `APS` | 计划需求、MRP 结果 | 提供排程输入 | API / 事件 |
| `APS` | `MES` | 排程结果 | 承接执行计划 | 事件 + 批量下发 |
| `PLM` | `MES / ERP / APS / QMS` | BOM、工艺、标准版本 | 发布生效版本 | 版本事件 + 查询 |
| `MES` | `WMS` | 领料需求、完工入库请求 | 驱动库存动作 | 事件 / API |
| `WMS` | `MES` | 领料完成、到料、库存状态 | 通知现场约束变化 | 事件 |
| `MES` | `QMS` | 待检对象、过程异常 | 创建检验与裁决入口 | 事件 / API |
| `QMS` | `MES / WMS` | 检验结论、不合格裁决 | 控制放行、冻结、返工、报废 | 事件 |
| `IoT` | `MES` | 告警、停机、自动报工 | 驱动暂停、锁定、自动上报 | 实时事件 |

## 10.2 三个必须记住的集成边界

### 领料与消耗

- `WMS`：仓库拿了什么
- `MES`：现场用了什么

### 检验与裁决

- `QMS`：检验结论、特采、返工、报废裁决
- `MES / WMS`：执行被裁决后的动作

### 发布与切换

- `PLM`：版本合法性和发布
- `MES`：现场实际切换和旧版收口

## 十一、异常补偿与对账规则

## 11.1 重试规则

建议统一采用：

- 第 1 次：立即重试
- 第 2 次：5 秒后
- 第 3 次：30 秒后
- 第 4 次：5 分钟后
- 超过阈值：进入补偿池

## 11.2 补偿池最少字段

- 消息编号
- 来源系统
- 目标系统
- 对象类型
- 对象编号
- 失败时间
- 重试次数
- 失败原因
- 当前处理状态

建议状态：

- `pending_retry`
- `waiting_manual_fix`
- `replayed`
- `closed`

## 11.3 人工补偿原则

人工补偿不能直接改数据库，必须：

1. 先确认主责系统真相
2. 再重放事件或补录动作
3. 最后留痕关闭补偿任务

## 11.4 关键对账清单

| 对账主题 | 对账双方 | 核心检查项 |
| --- | --- | --- |
| 订单变更对账 | `CRM` vs `ERP / APS` | 订单版本、数量、交期 |
| 工单承接对账 | `APS` vs `MES` | 已发布排程是否都被承接 |
| 领料 / 消耗对账 | `WMS` vs `MES` | 领料数量、消耗数量、退料差异 |
| 检验 / 锁定对账 | `QMS` vs `MES / WMS` | 判退对象是否已冻结 / 隔离 |
| 版本切换对账 | `PLM` vs `MES` | 已发布版本与现场生效版本 |
| 停机影响对账 | `IoT` vs `MES` | 停机事件是否映射到受影响任务 |

## 十二、开发实施建议

## 12.1 页面设计原则

每新增一个页面，都必须先回答：

1. 它服务哪个核心对象？
2. 它处在哪个状态控制点？
3. 它触发哪个业务事件？
4. 它是否涉及审批放行？
5. 它是否跨系统联动？

如果这五个问题答不出来，这个页面大概率还是“功能点页面”，不是平台型页面。

## 12.2 后端接口设计原则

- 接口围绕对象设计，不围绕前端表格设计
- 状态变更接口优先显式化
- 审批动作和执行动作分离
- 必须设计幂等键
- 必须保留审计链

## 12.3 前端开发建议

- 菜单与路由必须与业务闭环一致
- 隐藏详情页要能高亮所属菜单
- 占位页不是废页，而是未来能力入口
- 状态标签、审批动作、异常动作应优先标准化

## 12.4 测试关注点

测试不能只测页面能不能点开，必须重点验证：

1. 状态是否按规则迁移
2. 锁定后是否真的不能继续执行
3. 审批通过后是否真正放行
4. 跨系统失败后是否进入补偿
5. 对账差异是否可发现

| 管理分析 / 合规与综合管理 | 安环、人资、制度执行 | 专业系统 / 平台聚合 | 记录、预警、检查，不侵入 MES 真相 |
| 平台基础 / 系统管理 | 用户、角色、菜单、审批、日志 | 平台基础层 | 权限模型、审批编排、审计链 |
| 平台基础 / 主数据治理 | 物料、客户、供应商、编码 | `MDM` | 标准口径、一处主责、多系统同步 |
| 平台基础 / 平台设置 | 参数、规则、通知、集成配置 | 平台基础层 | 规则参数化、可配置而非硬编码 |

## 十六、页面建设最小交付清单

后续无论建设新页面，还是补建“等待建设”占位页，建议都至少补齐以下内容：

### 16.1 页面定义最小项

1. 页面服务的核心对象
2. 页面所在菜单域
3. 页面主责系统
4. 页面触发的状态动作
5. 页面可能产生的业务事件
6. 页面涉及的审批点
7. 页面涉及的外部系统联动

### 16.2 列表页最小项

- 查询条件
- 状态标签
- 主操作按钮
- 批量操作规则
- 导出 / 打印 / 追溯入口
- 异常提示与空态提示

### 16.3 详情页最小项

- 头部对象状态区
- 基本信息区
- 生命周期时间轴
- 审批记录区
- 关联对象区
- 操作日志区

### 16.4 执行页最小项

- 当前任务上下文
- 前置条件校验结果
- 执行动作区
- 异常上报入口
- 锁定原因展示
- 恢复 / 放行结果展示

### 16.5 联调前必须确认

1. 主责系统是谁
2. 谁创建对象
3. 谁改状态
4. 谁发事件
5. 失败后谁补偿
6. 对账双方是谁

## 十八、结论

这份总册的目标，是让开发团队始终围绕四个关键词推进：

```text
对象
事件
状态
补偿
```

只要后续每个功能点都能回到这四件事上校验，平台就不容易在开发过程中重新滑回“页面很多、规则很散、系统很乱”的老路。
