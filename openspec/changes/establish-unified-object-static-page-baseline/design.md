## Context

当前项目已经完成了一轮菜单命名、路由拆分、页面归属和首页工作台方向的收敛，但系统下一阶段仍面临三个核心问题：

- 页面要继续扩展时，缺少统一的对象基线，容易再次出现“同一对象在不同模块重复定义”
- 静态页、mock、API 的交接顺序不明确，容易一边写页面一边补接口语义
- 虽然系统架构已明确覆盖 `MDM / PLM / APS / MES / WMS / QMS / ERP / SRM / CRM / BI`，但页面开发若不按对象与链路推进，仍会看不出一体化价值

本次变更的直接利益相关方包括产品、前端、后端和后续测试人员。它既是静态页开发规则，也是后续 mock、DTO 和接口边界的统一前置约束。

当前已有文档基础包括：

- `docs/integrated-data-governance-blueprint.md`
- `docs/master-data-object-catalog.md`
- `docs/core-data-field-standard.md`
- `docs/page-object-mapping-matrix.md`
- `docs/object-state-flow-matrix.md`
- `docs/page-development-priority-roadmap.md`

这些文档已经完成“思路沉淀”，本次设计的任务是把它们转为 OpenSpec 可实施变更。

## Goals / Non-Goals

**Goals:**

- 建立跨模块统一的核心对象基线，确保主责、字段和状态流有单一依据
- 明确静态页阶段必须交付的页面结构和数据边界，避免过早进入 mock 和 API 设计
- 建立第一阶段页面开发顺序，让一体化架构先从 `MDM + PLM` 对象定义页落地，再扩展到执行和协同链路
- 让后续每个页面都有可追溯的规范来源，减少前后端和产品之间的口径分歧

**Non-Goals:**

- 本次变更不直接实现真实接口联调
- 本次变更不引入新的集成平台或中间件
- 本次变更不覆盖全部业务模块的最终细节页面
- 本次变更不处理排程算法、库存结算算法或设备实时采集逻辑

## Decisions

### Decision 1: 采用“对象优先”而不是“模块优先”的静态页基线

系统后续开发将优先围绕物料、BOM、工艺路线、工单、库存事务、质检事务等核心对象组织页面，而不是先按 `MES`、`WMS`、`QMS` 各自扩页。

原因：

- 一体化系统的返工主要来自对象口径不统一，而不是页面数量不足
- 同一对象往往被多个模块消费，必须先统一对象真相
- 这样更符合文章中提出的数据治理主线

替代方案：

- 按模块分别推进页面：速度看似快，但会把字段、状态和交互差异固化到各模块，后续整合成本更高

### Decision 2: 静态页交付必须先于 mock，mock 必须先于 API

页面开发采用三段式交付：

1. 静态页阶段：完成结构、字段、状态、按钮、详情区块
2. mock 阶段：补齐列表、详情、状态流转、行项目结构
3. API 阶段：再落 DTO、校验和接口边界

原因：

- 当前用户已明确不希望一边写 mock 一边写页面
- 先稳定页面和对象结构，后续 mock 与 API 更容易统一
- 可以降低“前端页面形态尚未定型，后端数据结构先固化”的风险

替代方案：

- 页面与 mock 同时推进：短期看节省步骤，长期会把不成熟的数据结构提前固化

### Decision 3: 第一阶段只聚焦 `MDM + PLM` 对象页与最小执行链路入口

第一阶段优先完成：

- `MDM`：物料、仓库、客户、供应商等主数据页
- `PLM`：EBOM、MBOM、工艺路线、ECN 等定义页

第二阶段再进入：

- `APS/MES/WMS/QMS` 的工单、报工、领退料、入库、检验链路

原因：

- `MDM` 与 `PLM` 是几乎所有后续页面的上游基础
- 如果这些对象先不稳定，执行页只能继续使用临时字段和临时状态

替代方案：

- 直接先做工单、报工等执行页：容易在没有稳定产品/工艺/物料口径时产生二次返工

### Decision 4: 文档基线与页面实现解耦，但必须建立一一映射

`docs/` 下的治理文档不直接作为运行时配置读取，但每个页面实现、静态数据文件、mock 结构和 DTO 设计都必须能映射回这些基线。

原因：

- 规范文档适合承载抽象规则和跨页约束
- 运行时代码仍应保持清晰、轻量和面向实现
- 通过映射关系可避免文档与代码完全脱节

替代方案：

- 直接把文档结构转为代码配置中心：前期成本高，且当前阶段规则仍在收敛，不适合立即做成运行时抽象

## Risks / Trade-offs

- [风险] 规范完整但页面团队执行时再次各自发挥  
  → Mitigation：后续每一批页面任务必须引用本次变更中的对象、字段、状态和优先级规范

- [风险] 范围过大，导致迟迟不进入真正页面交付  
  → Mitigation：本次变更只要求形成一期基线，不要求一次覆盖全模块全部页

- [风险] `ERP/SRM/CRM/BI` 等经营协同域细节尚未完全展开  
  → Mitigation：先把这些域放入路线图和依赖关系，不在本次变更中深挖次级对象

- [风险] 文档基线与现有页面残留实现不一致  
  → Mitigation：后续页面重构按批次推进时，必须先做页面-对象映射校验

## Migration Plan

1. 以本次变更作为新的静态页开发准入基线
2. 第一批任务先落到 `MDM` 和 `PLM` 核心对象页
3. 页面静态结构稳定后，再按相同对象模型补 mock 响应
4. mock 结构稳定后，再由后端设计 DTO 和接口边界

如果后续实施中发现部分对象定义不完整，应回到本次变更补充对象要求，而不是让单个页面私自扩展对象口径。

本次变更本身不涉及运行时发布，因此不存在传统意义上的生产回滚动作；其“回滚”方式是撤回未落地的页面规范或重新修订对应 spec。
