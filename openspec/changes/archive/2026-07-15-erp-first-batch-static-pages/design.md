## Context

当前项目主链已经按 `MDM -> PLM -> MES -> WMS -> QMS -> APS` 的顺序推进，`APS` 首批静态页也已经正式归档。接下来按既有基线应进入 `ERP`，把计划承接与核算承接页面建立在已稳定的上游对象之上，而不是再回头重定义需求、排程或执行真相。

当前仓库中的 `ERP` 代码处于“已有页面，但批次边界未正式冻结”的中间态：

- `src/router/modules/erp/index.ts` 已经挂出 `需求池`、`MRP` 链和 `财务` 链页面
- `需求池` 仍然是 `plannedPage` 占位，没有进入正式静态页批次
- `MRP` 结果、历史、预测、多工厂、净变更都已经有视图，但还没有通过正式 OpenSpec 约束哪个属于首批承接页、哪个属于后续扩展页
- `finance/index` 路由标题是“财务概览”，页面实际却是“应付管理列表”，说明页面命名、对象归属和批次定位仍有漂移
- `finance/report`、`finance/cost` 更偏分析视角，如果在首批就与 `需求池 / MRP / 总账` 并列验收，会冲淡 ERP 首批“计划承接 + 核算承接”的主线

这次设计的目标不是进入真实接口联调，而是先把 `ERP` 首批静态页的范围、对象边界、页面拆分和壳层规则冻结下来，让后续实现、mock 和 API 设计继承同一套口径。

## Goals / Non-Goals

**Goals:**

- 冻结 `ERP` 首批静态页的正式范围与推荐顺序
- 冻结 `需求池`、`MRP` 结果链、`总账对账` 的主对象边界
- 冻结 `ERP` 与 `APS`、`CRM`、`MES`、`WMS` 之间的消费/协作边界
- 冻结本批页面只交付静态页面结构、静态字段、状态表达、默认跳转关系与稳定静态数据
- 冻结结果页、CRUD 页、台账页在 ERP 模块中的页面壳层规则
- 为后续实现阶段提供可以直接执行的任务拆分基础

**Non-Goals:**

- 本次变更不引入真实 MRP 算法、排程求解器或财务核算引擎
- 本次变更不进入真实 API 联调，也不以接口可用性作为验收前提
- 本次变更不把 `预测需求`、`多工厂计划`、`成本分析`、`财务报表`、`财务概览` 作为首批必须验收页面
- 本次变更不重定义 `CRM` 的销售订单真相、`APS` 的排程真相、`MES` 的执行真相或 `WMS` 的库存真相

## Decisions

### Decision 1: ERP 首批范围固定为“需求池 + MRP 结果链 + 总账对账”

本次正式纳入交付范围的页面固定为：

- `需求池`
- `物料需求结果`
- `物料需求历史`
- `净变更计划`
- `总账对账`

明确排除在首批验收之外的页面包括：

- `预测需求`
- `多工厂计划`
- `财务概览`
- `成本分析`
- `财务报表`

这样可以先稳定 ERP 最贴近制造主链的“计划承接批”，即：

```text
CRM/业务需求
  -> ERP 需求池
  -> ERP MRP 结果/历史/净变更
  -> ERP 总账承接
```

备选方案是把现有 `ERP` 路由下所有页面一起并入首批，但那会把“承接页”和“分析页”混成一个批次，导致 acceptance 口径过宽，因此不选。

### Decision 2: 需求池必须作为 ERP 首批的正式主入口，而不是继续停留在 plannedPage

`需求池` 的主对象固定为 `计划需求`。它表达的是 ERP 对销售需求、预测需求、计划释放输入的承接与汇总，而不是：

- `CRM` 销售订单维护页
- `APS` 排程结果页
- `SRM` 采购申请页

因此该页应优先使用标准 CRUD/查询页语义：

- 搜索区
- 列表区
- 状态标签
- 优先级/合并/释放准备动作
- 与 `MRP` 结果链的默认跳转关系

备选方案是继续把 `需求池` 留作占位页，先围绕已有 MRP 页面起变更；但这会使 ERP 首批缺少真正的计划输入对象入口，后续对象链条不完整，因此不选。

### Decision 3: MRP 结果链只表达 ERP 建议承接，不表达采购执行或 MES 执行真相

`物料需求结果`、`物料需求历史`、`净变更计划` 在本批统一归入 “MRP 结果链”，其边界固定为：

- `ERP` 拥有建议采购、建议生产、历史运行、变更影响的真相
- `CRM` 继续拥有销售订单与原始客户需求真相
- `APS` 继续拥有排程结果与重排建议真相
- `MES` 继续拥有工单、任务、执行状态真相
- `WMS` 继续拥有库存、批次、收发存事实真相

这意味着：

- `MRP` 结果页可以出现“生成采购申请”“生成工单”等动作，但动作表达的是 ERP 建议承接，不是执行完成
- `净变更计划` 与 `APS` 的“扰动重排”不能混页，前者是物料/供需变化重算，后者是排程建议调整
- `MRP` 历史只表达 ERP 侧运行记录与版本回看，不扩展成 BI 分析报表

备选方案是把 `MRP` 结果直接设计为采购执行或工单执行入口，但这会把 ERP 结果真相与下游执行真相混写，因此不选。

### Decision 4: 总账对账是 ERP 首批唯一财务承接页，分析类财务页后置

首批财务页只纳入 `总账对账`，主对象固定为：

- `总账分录`
- `借贷汇总`
- `对账结果`

它表达的是 ERP 对上游业务事实的核算承接，而不是上游业务对象的维护：

- 不是 `WMS` 库存台账页
- 不是 `MES` 报工或完工页
- 不是 `BI` 经营分析页

同时这一定义也反向约束现有财务页面：

- `finance/index` 与“财务概览”的命名漂移需要在实现阶段收口
- `finance/report`、`finance/cost` 暂不进入首批验收

备选方案是把财务概览、报表、成本分析一起并入首批，以形成完整 ERP 财务链；但当前更需要的是稳定核算承接对象，而不是提前做分析层，因此不选。

### Decision 5: ERP 首批页面壳层按“CRUD 输入页 / 结果页 / 台账页”区分

本批页面的壳层规则固定如下：

- `需求池` 优先采用 `CrudPage` 或等价的标准 CRUD 容器
- `物料需求结果`、`物料需求历史`、`净变更计划` 属于结果链页面，允许保留运行摘要、结果标签页、建议动作区和影响块，但不得引入新的整页壳层体系
- `总账对账` 属于查询/台账页，允许保留借贷卡片、分录表、对账表，但不得膨胀为经营分析大屏

这样既能保留 ERP 页面差异化表达，又能维持统一页头、动作区、静态数据组织和跳转结构。备选方案是把所有 ERP 页面全部压成纯 CRUD 页，但会削弱结果链和台账页的业务表达，因此不选。

## Risks / Trade-offs

- [Risk] 现有 `ERP` 页面已经存在多种命名和页面语义，正式收口时会出现“当前页面内容”和“首批对象定义”之间的过渡成本
  - Mitigation: 先冻结对象边界和批次规则，再在实现阶段分步调整路由标题、页面标题和页面主对象
- [Risk] `需求池` 当前仍是占位页，起步时会比其它 ERP 页面多一步“从 plannedPage 变正式页”的成本
  - Mitigation: 在任务中把 `需求池` 作为首个实现对象，先完成主入口，再带动 MRP 结果链统一
- [Risk] `finance/index` 的“财务概览 / 应付管理”语义漂移如果不收口，会继续污染 ERP 首批边界
  - Mitigation: 在设计中明确首批财务页只验收 `总账对账`，并把 `finance/index` 作为需要后续改名或移批的页面处理
- [Risk] MRP 结果页的动作按钮容易被误实现成“执行完成”而不是“建议承接”
  - Mitigation: 在规格中显式约束 ERP 只拥有建议和承接真相，不拥有采购、工单或库存执行真相

## Migration Plan

1. 创建 `ERP` 首批静态页正式 OpenSpec 变更
2. 新增 `erp-first-batch-static-pages` capability
3. 在 roadmap 和 ownership 能力下补充 ERP 增量规格
4. 后续实现阶段先从 `需求池` 开始，补齐正式静态页与静态数据结构
5. 再统一 `MRP` 结果链页面的主对象、动作层级和默认跳转
6. 最后收口 `总账对账` 页面，并把分析类财务页明确后置
7. 静态页验收通过后，再决定是否进入 mock 阶段或进一步展开财务分析页

## Open Questions

- `需求池` 第一轮更适合做成单页列表页，还是“列表 + 汇总卡片 + 释放准备侧栏”的组合页
- `物料需求历史` 第一轮是保留独立路由页，还是先作为 `MRP` 结果页的稳定跳转子页
- `finance/index` 后续更适合改名为“应付管理”并移出首批，还是回归真正“财务概览”的聚合角色
