## Context

当前 `BI` 首批已经稳定为 `生产分析看板`、`质量分析看板`、`库存分析看板`、`经营分析看板`，并通过 `AnalysisSnapshotPage` 与 `src/static/services/bi.ts` 建立了统一的静态分析快照结构。二批页面需要承接首批延后的专题入口：能耗、安环、人资和报表，但仍处于静态页阶段。

本次实现继续把 `BI` 定位为分析与目录展示层。能耗、安环、人资页面展示跨系统聚合后的专题分析快照；报表管理展示可用报表目录和来源关系，不提供报表设计器、权限指标建模、真实导出、调度任务或数据仓库接入。

## Goals / Non-Goals

**Goals:**

- 新增 `能耗分析`、`安环分析`、`人资分析` 三个专题分析静态页
- 新增 `报表管理` 静态目录页，展示报表名称、主题、更新周期、来源系统、状态和负责人
- 复用 `AnalysisSnapshotPage`、`gi-page-layout`、`CrudPage` 等现有页面基线
- 扩展 `src/static/services/bi.ts`，让首批与二批分析页共享同一快照数据结构
- 更新 `BI` 路由菜单顺序，把二批专题页放在四个首批核心看板之后

**Non-Goals:**

- 不实现真实 API、mock 接口迁移、OLAP、数据仓库、指标权限、报表设计器或导出后端
- 不删除旧版隐藏页面，避免扩大迁移风险
- 不在运行页展示所有权说明、边界说明或产品介绍块
- 不将能耗、安环、人资源业务事实迁移到 `BI`

## Decisions

### 1. 专题分析继续复用快照组件

`能耗分析`、`安环分析`、`人资分析` 与首批看板一样，主交互是周期筛选、指标卡、趋势、结构分布和来源明细。复用 `AnalysisSnapshotPage` 能保持页面节奏一致，并让后续 mock / API 继承同一字段结构。

备选方案：

- 为每个专题单独写定制图表页
  - 放弃原因：二批仍是静态页阶段，过早定制会增加口径分歧
- 恢复旧版能耗、安环、人资多级页面
  - 放弃原因：旧页面颗粒度分散，不符合当前“先稳定导航与静态结构”的节奏

### 2. 报表管理使用列表型工作页

`报表管理` 的主对象是 `报表目录`，更接近维护和查阅型列表，而不是图表快照。因此使用 `CrudPage` 承载搜索、表格、分页、刷新和行操作，用静态服务提供数据。

备选方案：

- 使用分析快照组件展示报表
  - 放弃原因：报表目录不是趋势分析对象，强行复用会混淆主对象
- 做完整报表设计器
  - 放弃原因：当前阶段没有进入指标建模、模板设计、权限和调度能力

### 3. 二批路由开放新聚合入口，旧页面继续隐藏

新路由使用 `energy-analysis`、`ehs-analysis`、`hr-analysis`、`report-catalog` 作为稳定入口。旧 `energy/**`、`ehs/**`、`hr/**` 和 `ReportView.vue` 保留隐藏，便于后续对照迁移。

备选方案：

- 直接复用旧路由路径
  - 放弃原因：旧路由命名和页面粒度不利于二批聚合入口表达
- 删除旧页面
  - 放弃原因：会扩大本次变更范围，且不是静态二批验收必要条件

### 4. BI 保持消费源系统真相

能耗分析消费 `IOT / EAM / MES` 的采样、设备和生产上下文；安环分析消费 `QMS / MES / EAM` 等安全、环境、培训和异常上下文；人资分析消费人员、班次、技能、考勤和计件相关事实；报表目录只记录报表展示和组织信息。`BI` 不维护这些源事实。

## Risks / Trade-offs

- [Risk] 二批专题容易扩散成多级功能树 -> Mitigation：本次只开放四个聚合入口，旧细分页继续隐藏
- [Risk] 报表管理被误解为完整报表平台 -> Mitigation：页面命名和任务限定为报表目录，排除设计器、调度、导出后端
- [Risk] 源系统字段口径不稳定 -> Mitigation：静态数据中保留 `source_systems` 与来源明细，明确 BI 消费上游事实
- [Risk] 复用快照组件限制专题差异表达 -> Mitigation：二批先稳定导航和静态结构，后续需要时再在新变更中拆分专题可视化
