## Context

根据 `docs/page-object-mapping-matrix.md`，`plm/bom/cost` 属于 `BOM + 物料成本` 的派生分析页，主责在 `PLM / ERP` 边界之间：

- `PLM` 提供结构、用量、路线和变更真相；
- `ERP` 提供最终成本口径与财务核算真相。

因此这张页面不应表现成 `ERP` 财务报表，也不应停留在只有一个下拉框和一张简单表格的工具页。

## Goals / Non-Goals

**Goals**

- 让 `结构成本估算` 进入统一静态页骨架。
- 固定成本估算页的查询区、概览卡片、明细表、成本构成图和详情结构。
- 明确 `PLM` 与 `ERP` 的成本边界。
- 为后续 `ECN` 评估、版本切换、工艺调整提供统一的结构成本视图。

**Non-Goals**

- 本次不接入真实 `ERP` 成本接口。
- 本次不实现最终财务成本核算与会计凭证逻辑。
- 本次不与采购价格中心做联动。

## Decisions

### Decision 1: 成本页属于 `PLM` 派生分析，不等于 `ERP` 最终核算

页面要明确说明：这里是基于结构、工时和制造费用参数的静态估算，供工程和工艺评估使用；最终财务口径仍由 `ERP` 负责。

### Decision 2: 成本页必须同时表达结构明细与成本构成

用户不只需要看到总成本，还要看到：

- 哪些结构项成本最高；
- 材料 / 人工 / 制造费用如何分布；
- 变更单、路线版本和替代策略如何影响估算结果。

### Decision 3: 详情抽屉固定版本与影响区块

明细行详情在静态阶段就要固定：

- 结构节点基础信息；
- 成本拆分；
- 版本与路线来源；
- 变更影响与下游消费关系。

## Risks / Trade-offs

- 风险：用户可能把估算页误认为财务结算结果。  
  缓解：在页面归属说明和边界文案里明确“估算”与“最终核算”的分工。

- 风险：本次静态数据不能覆盖全部成本场景。  
  缓解：先固定页面结构，后续接口阶段继承同一字段区块。

## Migration Plan

1. 创建新的 OpenSpec 变更文档，锁定范围为 `PLM` 结构成本估算页。
2. 在 `PLM` 路由中补齐页面归属元信息。
3. 重写 `src/views/plm/bom/cost/index.vue` 为统一静态分析页。
4. 运行 lint、类型检查和构建验证。
