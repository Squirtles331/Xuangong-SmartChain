## Context

当前项目已经按 `MDM -> PLM -> MES -> WMS -> QMS -> APS -> ERP` 的顺序推进，`ERP` 静态页也已完成归档。根据现有开发基线，下一步不应跳转到分析型模块，而应进入 `SRM`，先把供应侧采购协同主链稳定下来，再继续进入 `CRM`。

当前仓库中的 `SRM` 代码处于“已有页面，但批次边界和页面基线尚未正式冻结”的状态：

- `src/router/modules/srm/index.ts` 已经挂出 `采购申请`、`采购订单`、`采购退货`、`价格协议`、`供应商协同`
- `src/views/srm/**` 页面当前仍主要使用 `gi-page-layout`
- 第一批、第二批页面尚未通过正式 OpenSpec 固定范围和边界
- `SRM` 与 `MDM`、`ERP`、`WMS`、`QMS` 的真相分工还没有在静态页层面完全收口

这次设计的目标不是立即推进接口，而是先把 `SRM` 第一批静态页的范围、静态数据口径、页面壳层和协同边界稳定下来，让后续实现、mock 和 API 工作继承同一套规则。

## Goals / Non-Goals

**Goals:**

- 冻结 `SRM` 第一批静态页的正式范围与推荐顺序
- 冻结 `采购申请`、`采购订单`、`采购退货` 的主对象边界
- 冻结 `SRM` 与 `MDM`、`ERP`、`WMS`、`QMS` 的采购协同边界
- 冻结第一批页面只交付静态页面结构、静态字段、状态表达、默认跳转关系和稳定静态数据
- 冻结第一批页面统一使用共享 CRUD 页面基线
- 为后续实现阶段提供可直接执行的任务拆分

**Non-Goals:**

- 本次变更不引入真实供应商门户认证、消息推送或外部协同接口
- 本次变更不把 `价格协议`、`供应商协同` 纳入第一批正式验收范围
- 本次变更不进入真实 API 联调，也不以接口可用性作为验收前提
- 本次变更不重定义 `MDM` 供应商主数据真相、`ERP` 计划和财务承接真相、`WMS` 收货与库存事务真相、`QMS` 来料质量裁决真相
- 本次变更不强行绑定 `SRM` 路由重构或目录重构为单独子变更

## Decisions

### Decision 1: SRM 第一批范围固定为“采购申请 + 采购订单 + 采购退货”

本次正式纳入交付范围的页面固定为：

- `采购申请`
- `采购订单`
- `采购退货`

明确排除在第一批验收之外的页面包括：

- `价格协议`
- `供应商协同`

这样可以先稳定 `SRM` 最贴近制造主链和 ERP 承接链的采购协同主线，即：

```text
ERP 采购建议 / 业务来源
  -> SRM 采购申请
  -> SRM 采购订单
  -> SRM 采购退货
  -> WMS 收货 / QMS 来料 / ERP 应付承接
```

备选方案是把当前 `SRM` 全部页面一次性纳入第一批，但会把“采购协同主链”和“价格/门户扩展页”混成一个批次，导致验收范围过宽，因此不选。

### Decision 2: 第一批页面统一采用共享 CrudPage 基线

`采购申请`、`采购订单`、`采购退货` 都属于标准的搜索/列表/工具栏/行操作/对话框页面，因此统一采用 `CrudPage` 作为默认页面壳层。

允许保留的扩展块包括：

- 采购申请到采购订单的转换块
- 采购订单中的收货关联或到货跟踪块
- 采购退货中的来源单据、质量原因和供应商反馈块

但这些扩展都必须建立在共享 CRUD 基线上，而不是重新引入单页自定义壳层体系。

备选方案是继续保留 `gi-page-layout` 并逐页微调，但这会与 ERP 刚收口出来的页面基线继续分叉，因此不选。

### Decision 3: SRM 第一批静态数据集中到共享静态服务层

第一批页面的数据结构统一沉淀到 `src/static/services/srm.ts` 或等价的 SRM 静态服务层，而不是继续散落在页面本地或历史 mock 结构里。

第一批需要稳定的核心数据包括：

- 采购申请对象
- 采购订单对象
- 采购退货对象
- 供应商响应与交付承诺字段
- 来源单据、关联收货、关联质量信息字段
- 状态标签和默认动作字段

这样可以让后续 mock、API DTO 和跨页面跳转统一继承同一套字段口径。

### Decision 4: SRM 页面只拥有采购协同真相，不重写上下游事务真相

本次明确 `SRM` 的主对象为采购协同对象，而不是全链路事务真相拥有者：

- `MDM` 继续拥有供应商主数据、分类和基础资质真相
- `ERP` 继续拥有采购建议来源、应付承接和财务携带真相
- `WMS` 继续拥有收货、退货入出库和库存事务真相
- `QMS` 继续拥有来料检验、质量裁决和不合格决定真相
- `SRM` 只拥有供应商协同状态、采购沟通与协同订单对象

这意味着：

- `采购申请` 可以消费 ERP 建议来源，但不能改写 ERP 计划真相
- `采购订单` 可以展示到货或收货关联信息，但不能把页面定义成 WMS 收货事务页
- `采购退货` 可以展示质量来源和退货原因，但不能把退货页定义成 QMS 裁决页或库存修正页

### Decision 5: 本变更先稳定静态页，不捆绑 SRM 路由重构

虽然当前 `SRM` 路由仍是平铺 children，但本变更优先解决的是第一批页面静态基线、对象边界和 CRUD 统一问题。

因此本次允许：

- 在现有 `src/router/modules/srm/index.ts` 下对标题、顺序、ownership metadata 做必要收口
- 暂不强制把 `SRM` 路由和目录一并升级成 ERP 那种新分组结构

备选方案是把“第一批静态页稳定”和“SRM 菜单/目录重构”绑成一个提案，但会增加范围和实现风险，因此当前不选。

## Risks / Trade-offs

- [Risk] `采购订单` 页面容易被实现成 ERP 单据页或 WMS 收货页
  - Mitigation: 在规格中显式约束 `SRM` 只拥有供应商协同真相，收货与财务只作为关联上下文

- [Risk] `采购退货` 页面容易把质量裁决和库存处理混成一个对象
  - Mitigation: 在规格中明确 `QMS` 负责裁决真相，`WMS` 负责库存事务真相，`SRM` 仅负责对供应商的退货协同对象

- [Risk] 第一批排除了 `价格协议` 和 `供应商协同`，模块看起来仍然“不完整”
  - Mitigation: 在 proposal 和 roadmap delta 中明确这是“第一批采购协同主链”，后续第二批再承接扩展页

- [Risk] 当前 `SRM` 页面全部还是旧壳层，统一切换到 `CrudPage` 时可能出现交互表达压缩
  - Mitigation: 保留转换、收货关联和退货原因等扩展块，但全部放入共享 CRUD 壳层的 slot 体系里

## Migration Plan

1. 创建 `SRM` 第一批静态页正式 OpenSpec 变更
2. 新增 `srm-first-batch-static-pages` capability
3. 在 `phase-one-page-roadmap` 和 `page-ownership-alignment` 下补充 `SRM` 增量规格
4. 后续实现阶段先沉淀 `src/static/services/srm.ts` 静态数据结构
5. 先完成 `采购申请`，再推进 `采购订单`，最后稳定 `采购退货`
6. 第一批静态验收完成后，再决定是否进入 `价格协议 / 供应商协同` 第二批或继续推进 `CRM`

## Open Questions

- `采购申请` 第一轮是否需要显式表达“由 ERP 建议转换而来”和“手工发起”两类来源
- `采购订单` 第一轮是否要保留到货/收货关联的扩展块，还是先只做纯协同订单页
- `采购退货` 第一轮是否需要展示来料检验结论摘要，还是只保留来源引用字段
