## Context

项目级开发顺序已经明确固定为：

`MDM -> PLM -> MES -> WMS -> QMS`

当前状态下，`MES` 已经完成推荐批次，下一步不应直接跳入 `QMS`、`APS` 或更后面的经营协同模块，而应先把 `WMS` 第一批仓储事务页稳定下来。现有代码里已经存在 `WMS` 路由和页面骨架，但这些页面还没有被正式定义为“静态阶段的正式交付批次”，也还没有统一锁定：

- 本批 `WMS` 页面到底包含哪些页面
- 页面现在是否只允许使用静态数据
- 哪些页面默认应沿用 `CrudPage`
- `MES -> WMS -> QMS` 的边界在页面层如何表达

这次变更的目标不是直接进入接口开发，而是把 `WMS` 第一批页面的结构、对象边界、静态数据组织方式和前端壳层约束固定下来，为后续实现和验收提供统一依据。

## Goals / Non-Goals

**Goals:**

- 冻结 `WMS` 第一批静态事务页的正式范围和顺序
- 冻结本批页面只使用静态数据、不进入真实 `API` 的阶段边界
- 冻结 `WMS`、`MES`、`QMS` 在本批页面中的对象主责和协同边界
- 冻结标准事务列表页优先使用 `CrudPage` 的前端基线
- 冻结本批页面的工具栏、行操作、状态标签、默认跳转和桥接表达

**Non-Goals:**

- 本次变更不引入真实接口联调
- 本次变更不推进复杂 mock 协议设计
- 本次变更不覆盖 `WMS` 条码、批次隔离、IoT 仓储感知等扩展页的正式交付
- 本次变更不提前进入 `QMS` 的不合格处置、特采放行、返工裁决等页面
- 本次变更不重写全局 CRUD 组件库，只复用当前 `CrudPage` 基线

## Decisions

### Decision 1: 第一批 `WMS` 页面范围只覆盖六张核心事务页

本次变更正式纳入交付范围的页面为：

- `收货入库`
- `生产领料`
- `库存台账`
- `退料退货`
- `库存盘点`
- `库存调拨`

以下页面暂不作为本批正式验收范围：

- `销售出库`
- `倒冲`
- `批次条码`
- `扫码作业`
- `批次隔离`

这样可以把本批聚焦在“仓储主事务链”本身，而不是过早把条码、扩展工具和质量隔离能力混在第一轮范围里。

### Decision 2: 静态阶段统一采用“页面结构 + 静态数据服务”模式

本批页面在当前阶段只允许使用静态数据，不进入真实 `API` 联调，也不要求先完成 mock 协议。

页面层允许继续保留现有调用口，但数据来源的正式方向应固定为：

- 页面结构在 `src/views/wms/**`
- 静态数据服务在 `src/static/services/wms.ts`
- 页面优先从静态服务读取列表、摘要和弹窗所需数据

这样做的原因是避免当前阶段继续把页面壳层和接口抽象耦合在一起，同时保留后续进入 mock / API 时的替换空间。

### Decision 3: 标准事务列表页默认使用 `CrudPage`

本批页面大多数属于标准事务列表页，页面结构均应优先收敛到：

- 搜索区
- 工具栏
- 列表表格
- 行操作
- 对话框或抽屉表单

因此，本批 `WMS` 页面默认采用 `CrudPage` 作为页面级容器。只有当页面明显演化为：

- 详情页
- 执行页
- 结构页

时，才允许不套用 `CrudPage`。

这个决定的目的是让 `WMS` 第一批页面在视觉和交互上与当前项目 CRUD 基线保持一致，而不是每页都重新拼装一套壳层。

### Decision 4: `WMS` 只维护库存事务真相，不吞并 `MES` 或 `QMS` 主责

本批页面必须保持以下对象边界：

- `WMS` 维护收货、领料、退料、盘点、调拨、库存余额、库存批次等事务真相
- `MES` 作为来源对象出现，例如工单领料、完工入库、倒冲来源
- `QMS` 只在后续裁决联动中提供质量决定，不改写库存事务真相

因此：

- `生产领料` 不能等同于 `MES` 实际投料和消耗
- `收货入库` 不能提前承担完工判定或质量放行结论
- `库存盘点` 只能表达库存调整事实，不能代替质量处置闭环

### Decision 5: 本批优先补齐 `MES -> WMS` 的桥接表达，而不是先扩展外围页

第一批 `WMS` 静态页不仅要能单页展示，还必须把与 `MES` 的关系表达清楚。当前阶段优先补齐三类桥接信息：

- 领料申请/领料执行
- 完工入库确认
- 倒冲记录

即使部分桥接页本身不纳入当前正式验收，也必须在字段、关系块、跳转口径上留下稳定表达，避免后续返工。

## Risks / Trade-offs

- 风险：当前已有部分 `WMS` 页面接在 `api/wms` 上，团队可能误以为已经进入接口阶段
  - 缓解：在本次变更中明确“当前阶段只认静态数据策略”，把接口联调排除出本批目标
- 风险：如果把过多扩展页一起纳入本批，`WMS` 第一轮范围会再次失控
  - 缓解：只冻结六张核心事务页，把条码、隔离、IoT 等扩展能力放到后续批次
- 风险：`CrudPage` 被过度滥用到复杂执行页，可能压扁页面表达
  - 缓解：本次明确 `CrudPage` 只是标准列表事务页默认壳，不是所有页面的强制模板
- 风险：`生产领料` 与 `MES` 投料消耗边界不清时，页面字段容易混乱
  - 缓解：在规格中单独写明领料事务与生产消耗事实不是同一对象

## Migration Plan

1. 创建 `WMS` 第一批静态事务页的正式 OpenSpec change
2. 增加新的 `wms-first-batch-static-pages` capability
3. 同步补充路线图与页面主责边界的增量规格
4. 在实现阶段统一将本批页面收敛到静态数据来源与统一页面壳层
5. 静态验收通过后，再决定是否进入 mock，再进入 API

## Open Questions

- `销售出库` 是否在 `WMS` 第一批结束后立即纳入下一轮，还是与后续 `ERP / CRM` 交付节奏联动
- `倒冲` 是否在下一轮仍放在 `WMS` 菜单内表达，还是以 `MES` 来源页加 `WMS` 事务块的方式承接
- `批次隔离` 是否应在 `QMS` 关键裁决页稳定后再正式进入实现
