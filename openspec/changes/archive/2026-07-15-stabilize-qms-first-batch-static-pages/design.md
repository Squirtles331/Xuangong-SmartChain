## Context

项目当前主链顺序已经固定为：
`MDM -> PLM -> MES -> WMS -> QMS`

`WMS` 第一批静态页已经完成正式变更并进入已归档状态，当前仓库中的 `QMS` 代码则处于“有骨架但未成批”的中间态：

- 路由层已经挂出 `检验任务`、`检验模板`、`供应商质量` 和多个规划页
- 页面层已经存在 `template`、`inspection`、`supplier-quality` 三组目录
- `src/static/services/qms.ts` 目前只稳定了 `检验任务` 的静态服务
- `检验任务` 仍把来料、过程、完工检验混在统一入口里，尚未明确拆分来源对象
- `供应商质量` 更接近分析聚合页，依赖检验沉淀，不适合作为这一批的优先交付核心

这次变更的目标不是立刻进入接口开发，而是先把 `QMS` 第一批静态页的结构、边界、拆分方式和页面壳层冻结下来，使后续实现能直接继承这些约束。

## Goals / Non-Goals

**Goals:**

- 冻结 `QMS` 第一批静态页的正式范围和页面顺序
- 冻结三类检验页必须挂接业务来源对象的页面拆分方式
- 冻结 `QMS`、`MES`、`WMS` 在本批页面中的主责边界和裁决/执行边界
- 冻结本批页面只使用静态页面结构和静态数据服务的交付规则
- 冻结标准检验列表页与模板维护页优先复用 `CrudPage` 的前端壳层基线
- 冻结 `不合格处置` 作为质量裁决入口的独立表达方式

**Non-Goals:**

- 本次变更不引入真实接口联调
- 本次变更不推进复杂 mock 协议设计
- 本次变更不把 `供应商质量` 纳入第一批正式验收范围
- 本次变更不提前实现 `特采放行`、`返工裁决`、`报废裁决`、`复检与关闭`
- 本次变更不改变 `MES` 的返工执行主责，也不改变 `WMS` 的库存事务主责

## Decisions

### Decision 1: 第一批 QMS 页面范围只覆盖“模板 + 三类检验 + 不合格处置”

本次变更正式纳入交付范围的页面为：

- `检验模板`
- `来料检验`
- `过程检验`
- `完工检验`
- `不合格处置`

以下页面暂不作为本批正式验收范围：

- `供应商质量`
- `特采放行`
- `返工裁决`
- `报废裁决`
- `复检与关闭`

这样可以先稳定质量链的基础对象、执行入口和裁决入口，再进入衍生分析页和更深的裁决闭环页，避免第一轮就把 QMS 扩成过宽范围。

备选方案是把 `供应商质量` 一并纳入本批，但它依赖检验沉淀结果，先做会把批次重心从“执行挂接页”偏到“分析聚合页”，因此不选。

### Decision 2: 统一检验任务页要演进为三类来源清晰的检验页

当前 `qms/inspection` 已经可以作为临时统一入口，但本批正式设计要求把检验表达拆成三类业务挂接页：

- `来料检验` 挂 `收货/到货`
- `过程检验` 挂 `工序执行`
- `完工检验` 挂 `完工入库/完工确认`

无论最终是拆成三个路由页，还是一个总入口下的三个稳定视图，都必须满足：

- 来源对象不同
- 字段分组不同
- 状态语义不同
- 默认跳转不同

不允许继续把三类检验长期作为单一泛化任务表处理，否则来源对象和后续裁决动作会混淆。

### Decision 3: QMS 页面保持“静态页面结构 + 静态数据服务”交付模式

本批页面在当前阶段只允许用静态数据完成页面交付，不要求真实 API，也不要求先完成复杂 mock DTO。

正式数据组织方向固定为：

- 页面结构在 `src/views/qms/**`
- 静态服务在 `src/static/services/qms.ts`
- `src/api/qms.ts` 继续作为模式切换入口，但静态验收以 static service 为准

这样做可以先稳定页面字段、判定结果、裁决动作和来源关联块，再把这些结构向 mock 和 API 继承，而不是边做页面边漂移数据口径。

### Decision 4: 标准检验页默认使用 CrudPage，但允许执行/裁决区块扩展

本批多数页面都属于标准的搜索、列表、工具栏、行操作、弹窗维护模式，因此：

- `检验模板`
- `来料检验`
- `过程检验`
- `完工检验`

默认应优先对齐 `CrudPage`。

同时，以下场景允许在 CRUD 基线之上扩展：

- 检验执行对话框
- 来源对象摘要块
- 不合格裁决结果块
- 明细项或检验项目表格

这样既保留统一壳层，又避免把执行与裁决页硬压成单纯表格 CRUD。

### Decision 5: 不合格处置必须独立表达“QMS 裁决”而不是“MES/WMS 执行结果”

`不合格处置` 在本批中不是附属弹窗，而是独立质量裁决页。它必须表达：

- 不合格来源检验单
- 问题等级与责任信息
- 裁决动作候选
- 是否返工/报废/特采/复检
- 与后续 `MES` 或 `WMS` 执行后果的关系

其中边界必须固定为：

- `QMS` 决定是否返工、报废、特采、复检关闭
- `MES` 负责返工执行链
- `WMS` 负责库存隔离、放行后的库存控制后果

备选方案是把裁决动作留在统一检验页内完成，但这样会弱化 `不合格处理单` 作为独立对象的存在，因此不选。

## Risks / Trade-offs

- [风险] 当前路由和页面目录仍以统一 `检验任务` 为中心，改批次后会出现“现有码结构”和“正式页面分批定义”之间的过渡期
  - Mitigation: 在 proposal/spec 中先冻结正式对象和顺序，后续实现可分步从统一入口演进为三类检验页
- [风险] 如果继续把 `供应商质量` 当成 QMS 第一批核心页，会把执行链页和分析页混在一起
  - Mitigation: 明确将其排除在本批正式验收之外，待检验链稳定后再单列推进
- [风险] `CrudPage` 可能被机械套用到执行和裁决页，压缩业务表达
  - Mitigation: 明确 `CrudPage` 仅作为默认壳层，允许执行明细、裁决块和来源摘要作为扩展区块
- [风险] `不合格处置` 若没有明确裁决/执行边界，后续容易把返工执行或库存放行动作误写进 QMS 主对象
  - Mitigation: 在 spec 中单独约束 `QMS` 只拥有裁决真相，执行后果仍回到 `MES/WMS`

## Migration Plan

1. 创建 `QMS` 第一批静态页正式 OpenSpec 变更
2. 增加新的 `qms-first-batch-static-pages` capability
3. 同步补充阶段路线和页面主责边界的增量规格
4. 后续实现阶段优先补齐 `src/static/services/qms.ts` 和 `QMS` 页面壳层统一
5. 静态验收通过后，再决定是否进入 mock 阶段，再进入 API 阶段

## Open Questions

- `来料检验` 最终是拆成独立路由页，还是保留在统一 `检验任务` 入口下以稳定分组视图表达
- `不合格处置` 在第一轮实现中是先以列表页 + 裁决弹窗交付，还是直接做成带明细区的独立事务页
- `供应商质量` 后续更适合归入 `QMS` 的第二批分析页，还是放到跨模块分析能力统一收口
