## Context

根据当前仓库里的玄工开发基线，`PLM` 第一批次不只是把 `BOM` 维护入口补齐，还要把围绕 `BOM` 的分析能力稳定下来，供下游执行和追溯页面直接继承。

已有页面里：

- `BOM列表` 已经具备静态优先结构；
- `结构版本比较` 仍是简单版本选择加差异表；
- `BOM展开/反查` 仍是输入编码后直接出表，没有归属说明、版本约束、节点详情和影响表达。

这使得 `BOM` 作为上游定义对象，虽然有了维护入口，但还缺少统一的分析视图。

## Goals / Non-Goals

**Goals**

- 让 `结构版本比较` 符合当前 `PLM` 静态页标准。
- 让 `BOM展开/反查` 能表达层级结构、引用关系和下游影响。
- 在页面和路由元信息中明确这两页由 `PLM` 主责，供下游消费。
- 为后续 mock 和接口阶段固定字段区块，不再临时拼装第二套结构。

**Non-Goals**

- 本次不接入真实接口或 mock 服务。
- 本次不重写 `BOM编辑` 页数据模型。
- 本次不推进成本分析页与图表联动重构。

## Decisions

### Decision 1: `结构版本比较` 不是工具弹窗，而是 `PLM` 的正式分析页

比较页应承接版本评审、变更评估和下游口径确认，而不是停留在两个下拉框加一张差异表的轻量工具形态。

原因：

- 它直接服务于 `ECN`、版本切换和制造口径对齐；
- 差异对象需要稳定地表达“新增 / 删除 / 修改 / 无变化”；
- 下游在领料、报工、替代料判断时需要消费同一套差异结构。

### Decision 2: `BOM展开/反查` 必须同时表达结构视角与引用视角

`BOM展开` 与 `反查引用` 不应拆成两个互不关联的轻表工具，而应共享统一查询条件和对象上下文。

原因：

- 用户本质上是在查看同一个 `BOM` 对象的“向下结构”和“向上引用”；
- 相同的版本、对象、状态和变更单信息应在两个视角中保持一致；
- 这样更适合后续承接库存影响、工单引用和替代关系。

### Decision 3: 分析页也要补齐对象归属和边界提示

`BOM` 分析页虽然不是直接的维护列表，但仍属于 `PLM` 定义链的一部分，必须明确：

- `PLM` 负责上游版本真相；
- `MES / WMS / QMS` 作为消费系统引用分析结果；
- 页面上的路线、变更和库存影响都是“被引用关系”，不是在本页重定义业务对象。

### Decision 4: 静态阶段固定详情区块

两张页面都要在静态阶段固定详情区块：

- `结构版本比较`：基础差异、数量/损耗变化、路线绑定影响、变更单来源、执行影响；
- `BOM展开/反查`：节点基础信息、供给方式、替代策略、版本来源、下游引用与库存影响。

原因：

- 后续 mock 和接口可直接继承；
- 避免不同页面再分别补出第二套字段结构。

## Risks / Trade-offs

- 风险：页面比早期工具页更重，初看会增加一些信息量。  
  缓解：通过概览卡片、状态标签和详情抽屉分层表达，保证首屏仍可快速扫描。

- 风险：本次仍是静态页，后续接口阶段需要把字段落到真实数据模型。  
  缓解：先把页面字段区块稳定下来，减少后续返工。

## Migration Plan

1. 创建新的 OpenSpec 变更文档，锁定范围为 `结构版本比较` 与 `BOM展开/反查`。
2. 补齐 `PLM` 路由中这两张分析页的归属元信息。
3. 重写 `结构版本比较` 页为静态优先分析页。
4. 重写 `BOM展开/反查` 页为静态优先分析页。
5. 运行构建验证，确认新的静态结构已可稳定承接后续阶段。
