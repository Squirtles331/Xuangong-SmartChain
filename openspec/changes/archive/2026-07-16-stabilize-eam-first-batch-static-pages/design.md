## Context

当前 `EAM` 已有 `设备台账`、`点检`、`保养`、`维修`、`备件`、`设备综合效率` 等旧版页面入口，但这些页面还没有按当前项目的统一静态页基线收口：页面容器不统一、静态数据层缺失、首批范围与后续扩展边界也没有被正式固化。与此同时，`CRM` 首批静态页已经完成，项目顺序已进入 `EAM` 阶段，因此需要先把设备管理主链稳定下来，再决定是否扩展到 `备件` 或分析类页面。

本次变更的目标不是覆盖完整 `EAM` 域，而是先把设备运维主链中的四个核心对象稳定下来：

- `设备资产`
- `点检计划 / 点检执行`
- `保养计划 / 保养执行`
- `维修工单 / 维修闭环`

它们分别对应设备台账、预防性点检、预防性保养和故障维修四类核心场景，也最适合沿用当前项目已经稳定下来的 `CrudPage + static service` 页面基线。

## Goals / Non-Goals

**Goals:**

- 将 `设备台账`、`点检`、`保养`、`维修` 正式定义为 `EAM` 首批静态页范围
- 将首批 `EAM` 页面统一到共享 `CrudPage` 基线
- 为首批页面建立稳定的 `src/static/services/eam.ts` 静态服务层，使后续 mock / API 能直接继承字段与状态语义
- 在路由元数据与规格中明确 `EAM` 首批页面的主对象、协同系统和真相边界
- 保持页面级交互聚焦于搜索、列表、状态标签、工具栏动作、行内动作和弹窗操作

**Non-Goals:**

- 本次不进入 mock payload 或 API DTO 设计
- 本次不实现 `备件` 与 `设备综合效率`
- 本次不把 `WMS` 的备件库存真相、`IoT` 的原始遥测真相或 `MES` 的执行真相迁移到 `EAM`
- 本次不引入新的页面壳、概览介绍区或统计卡片区

## Decisions

### 1. 首批范围固定为 `设备台账 / 点检 / 保养 / 维修`

虽然当前 `EAM` 菜单已经包含 `备件` 和 `设备综合效率`，但如果第一批同时打开，会把设备资产维护、备件库存协同和分析看板混在一起。先稳定四个运维主链页面，可以让后续 `备件` 与 `OEE` 继承统一对象口径。

备选方案：

- 一次性覆盖完整 `EAM`
  - 放弃原因：首批范围过宽，容易把库存真相和分析语义混入静态 CRUD 批次
- 先做 `备件`
  - 放弃原因：对象边界更容易与 `WMS` 库存交易真相交叉，不适合作为起始页

### 2. 页面级容器统一改为 `CrudPage`

这四个首批页面本质上都是标准的搜索 / 列表 / 工具栏 / 行操作 / 弹窗页面。统一到 `CrudPage` 后，能与已经完成的 `ERP`、`SRM`、`CRM` 批次保持一致，也能降低后续扩展的样式与结构分叉风险。

备选方案：

- 继续沿用现有旧页面壳
  - 放弃原因：会继续保留旧批次遗留结构，和当前基线不一致
- 为 `EAM` 单独设计一套设备管理页面壳
  - 放弃原因：当前阶段没有必要，会制造新的壳层分支

### 3. 静态数据先收口到 `src/static/services/eam.ts`

首批 `EAM` 页面需要稳定字段、状态、关联关系和跳转行为，但暂时不需要真实接口。将静态数据集中到 `src/static/services/eam.ts`，可以让 UI 与后续 mock / API 对接解耦，并保证四个页面共享一致的对象定义。

备选方案：

- 继续直接依赖旧 `@/api` 结构
  - 放弃原因：会让静态阶段绑定旧接口语义，难以统一字段口径
- 将静态数据散落在各页面内部
  - 放弃原因：不利于后续 mock / API 继承，也不利于跨页对象维护

### 4. 运行时页面不展示介绍区或统计卡片区

当前项目静态页基线已经明确不再把介绍区、归属说明区和统计卡片区作为首批静态页的必需结构。`EAM` 首批页面只保留实际工作结构，避免视觉噪音和模板化冗余。

备选方案：

- 在页面顶部保留设备管理介绍区
  - 放弃原因：与当前产品基线冲突，也不提供实际业务操作价值
- 在列表上方增加统计卡片
  - 放弃原因：会把分析表达混入 CRUD 主页面，弱化页面主对象

### 5. 四个页面各自只维护一个主对象

- `设备台账`：设备资产对象
- `点检`：点检计划 / 点检执行对象
- `保养`：保养计划 / 保养执行对象
- `维修`：维修工单对象

相关的 `MDM`、`MES`、`IoT`、`WMS` 信息只作为关联上下文消费，不在页面上被重新定义为本地主对象。

## Risks / Trade-offs

- [Risk] `设备台账` 容易滑向通用主数据台账页 -> Mitigation：规格中明确 `EAM` 拥有设备资产真相，`MDM` 仅提供共享参照字段
- [Risk] `点检` 与 `保养` 的边界可能混淆 -> Mitigation：规格中区分点检的状态确认语义与保养的计划执行语义
- [Risk] `维修` 页面容易吸收 `IoT` 故障告警或 `MES` 停机事件真相 -> Mitigation：明确 `EAM` 只拥有维修工单与闭环过程，告警和执行事件作为来源上下文展示
- [Risk] 后续 `备件` 可能反向影响首批对象定义 -> Mitigation：本次先保持备件完全出批，待 `WMS` 与 `EAM` 交界规则明确后再扩展
