## Context

当前项目主链已经从 `MDM -> PLM -> MES -> WMS -> QMS` 顺序推进到 `QMS` 完成阶段，接下来按基线进入 `APS / ERP` 批次。但仓库里的 `APS` 目前仍处于“局部页面已存在、批次边界未正式冻结”的状态：

- `src/router/modules/aps/index.ts` 已定义 `排程结果`、`排程约束`、`排程历史`、`扰动重排`
- `排程结果` 和 `排程约束` 已有页面骨架，但仍沿用较早的页面容器与数据组织方式
- `排程历史` 与 `扰动重排` 仍是占位页，尚未进入正式静态页批次
- 当前 `APS` 页面既有图形化结果页，也有配置维护页，若不先冻结规则，后续很容易把计划输出、约束维护、历史复盘和现场扰动处理混为同一类页面

这次变更的目标不是推进真实排程引擎或接口联调，而是先把 `APS` 第一批静态页的范围、对象边界、前端壳层、静态数据策略和跨模块责任边界正式定义清楚，让后续实现和归档有统一基准。

## Goals / Non-Goals

**Goals:**

- 冻结 `APS` 第一批静态页的正式范围和页面顺序
- 冻结 `排程结果`、`排程约束`、`排程历史`、`扰动重排` 四类页面的主对象边界
- 冻结 `APS`、`MES`、`ERP` 在本批页面中的计划输出、执行事实和业务承接边界
- 冻结本批页面只交付静态页面结构、静态字段、状态表达、视图块和静态数据行为的规则
- 冻结 `APS` 图形化页面与 CRUD 维护页共存时的前端壳层规则

**Non-Goals:**

- 本次变更不引入真实 APS 求解引擎、排程算法或外部优化依赖
- 本次变更不推进真实 API 联调或复杂 mock 协议设计
- 本次变更不把 `ERP` 需求池、MRP 承接页纳入 APS 第一批静态页验收范围
- 本次变更不把 `MES` 现场执行调整页纳入 APS 页面主对象
- 本次变更不重定义 `MES` 的执行真相或 `ERP` 的业务单据真相

## Decisions

### Decision 1: APS 第一批页面范围限定为“结果 + 约束 + 历史 + 扰动重排”

本次变更正式纳入交付范围的页面为：

- `排程结果`
- `排程约束`
- `排程历史`
- `扰动重排`

不在本批范围内的内容包括：

- `ERP` 需求池与 `MRP` 承接页
- `MES` 现场派工或执行调整页
- `BI` 计划绩效分析页

这样可以先稳定 `APS` 自身的计划对象和页面结构，再让 `ERP` 承接计划输出，避免过早把计划层与业务承接层混在一起。备选方案是把 `ERP` 的 `需求池 / MRP结果` 一起纳入，但那会把本批从“APS 第一批静态页”扩成“APS+ERP 计划链”，超出当前最小闭环，因此不选。

### Decision 2: 四类 APS 页面分别绑定不同主对象，不允许用统一“计划页”泛化表达

四个页面的主对象固定如下：

- `排程结果` 主对象是 `APS 排程结果`
- `排程约束` 主对象是 `APS 排程约束`
- `排程历史` 主对象是 `APS 排程版本 / 参数快照 / 冲突复盘`
- `扰动重排` 主对象是 `APS 重排建议`

因此页面必须分别表达：

- 结果页关注计划输出、资源负荷、冲突提示
- 约束页关注规则维护，不等同于主数据维护中心
- 历史页关注版本对比与回放，不等同于 BI 报表
- 扰动重排页关注建议分层和影响范围，不等同于 MES 执行页面

备选方案是把历史与扰动都挂在结果页内作为子标签，但这样会削弱四类对象的边界，也不利于后续单独演进，因此不选。

### Decision 3: APS 第一批继续保持静态页模式，静态数据优先于接口抽象

本批页面在当前阶段只交付：

- 静态页面结构
- 静态字段与状态表达
- 静态图形块和结果视图
- 静态列表、筛选和弹窗行为

正式数据组织方向固定为：

- 页面结构在 `src/views/aps/**`
- 页面静态数据优先放在 `src/static/services/aps.ts` 或等价 APS 静态服务层
- 如已有 `api/aps` 占位能力，静态验收仍以静态服务结构为准，不反向要求真实接口完成

这样可以先稳定计划字段和状态口径，再进入 mock 和 API 阶段，避免边做页面边漂移排程结果和约束结构。

### Decision 4: APS 页面壳层按“结果页可扩展、维护页优先 CRUD”区分

`排程约束` 本质是标准维护页，应优先对齐 `CrudPage` 基线，包括搜索、工具栏、表格、行操作和弹窗维护。

`排程结果`、`排程历史`、`扰动重排` 不是纯表格 CRUD 页，允许保留图形块、摘要卡、版本对比和建议面板，但必须遵守统一规则：

- 页头结构统一
- 主动作与次动作层级统一
- 静态数据组织方式统一
- 不引入新的全页壳层系统

备选方案是把结果页也强行压成标准 CRUD 壳层，但那会弱化甘特、负荷和复盘视图的业务表达，因此不选。

### Decision 5: 扰动重排必须明确“APS 给建议，MES 执行调整”的边界

`扰动重排` 在本批不是 `MES` 的执行页，而是 `APS` 的重排建议页。它必须表达：

- 扰动等级
- 影响范围
- 局部重排与全局重排建议
- 约束冲突提示
- 与 `MES` 执行承接的关系

其中边界固定为：

- `APS` 负责重排方案和建议结果
- `MES` 负责执行层的实际调整
- `ERP` 仅消费计划影响，不拥有重排建议真相

备选方案是把现场扰动直接并入 `MES` 例外中心，但那会让 APS 失去作为计划建议层的独立表达，因此不选。

## Risks / Trade-offs

- [Risk] `APS` 现有页面已经存在旧壳层与旧数据依赖，正式推进时会出现“现有页面样式”和“新批次基线”之间的过渡成本
  - Mitigation: 在变更中先冻结对象和壳层原则，后续实现时分步收敛已有页面而不是一次性重写
- [Risk] `排程结果` 图形化表达较强，如果没有明确边界，后续容易被继续扩展成跨模块驾驶舱
  - Mitigation: 在规范中明确它只表达 APS 结果，不承接 BI 或 MES 执行真相
- [Risk] `排程约束` 容易与 `MDM/PLM` 主数据维护混淆
  - Mitigation: 明确其维护的是 APS 约束规则，不重新定义物料、设备、人员主数据真相
- [Risk] `扰动重排` 容易被误实现成现场执行调度页
  - Mitigation: 在规范中单独约束“APS 建议 / MES 执行”的责任边界

## Migration Plan

1. 创建 `APS` 第一批静态页正式 OpenSpec 变更
2. 新增 `aps-first-batch-static-pages` capability
3. 同步补充路线图和页面归属边界的增量规范
4. 后续实现阶段优先稳定 `排程结果` 与 `排程约束`，再补 `排程历史` 与 `扰动重排`
5. 静态页验收通过后，再决定是否进入 mock 阶段与 `ERP` 承接阶段

## Open Questions

- `排程结果` 最终是保持多标签单页，还是拆成“结果总览 + 冲突清单”两个稳定子入口
- `排程历史` 第一轮实现更适合做版本列表页，还是直接做带差异对比块的复盘页
- `扰动重排` 第一轮是以建议列表页交付，还是以分级建议卡片加对话框交付
