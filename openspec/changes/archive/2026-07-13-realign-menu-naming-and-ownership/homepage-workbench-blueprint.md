# 工作台首页模块蓝图

## 1. 文档目的

本文档用于把“工作台首页重设计”从概念层推进到可直接支持静态页面开发的蓝图层。

它回答四类问题：

1. 首页到底服务谁
2. 首页到底应该放哪些模块
3. 每个模块需要什么静态数据字段
4. 每个模块应该把用户导向哪个业务入口

本蓝图默认服务于当前静态优先阶段：

- 先搭静态页面结构
- 先准备静态数据
- 后续再接 mock / API

---

## 2. 首页定位

工作台首页不是经营分析大屏，也不是某个系统的“子首页”。

首页统一定位为：

> 登录后的跨域行动工作台

它的第一目标是告诉用户：

- 现在要做什么
- 现在哪里被阻塞了
- 现在哪个入口最值得点进去

它的第二目标才是提供轻量健康度摘要：

- 平台整体是否稳定
- 执行链条是否顺畅
- 交付、OEE、库存、质量是否存在明显风险

---

## 3. 目标用户

当前首页第一版不做强角色分发，但设计时默认覆盖以下用户：

- 计划员
- 车间主管 / 班组长
- 生产跟单 / PMC
- 质量工程师
- 仓储协同人员
- 管理层查看者

设计原则：

- 第一版先做统一工作台
- 各角色都能从首页找到自己的下一步入口
- 后续再做角色化裁剪

---

## 4. 页面总结构

推荐首页信息架构：

```text
工作台首页
├─ A. 我的待办
├─ B. 今日异常与预警
├─ C. 执行总览
├─ D. 协同域摘要
├─ E. 快捷入口
└─ F. 经营摘要（轻量）
```

推荐首屏优先级：

```text
第一屏优先出现：
1. 我的待办
2. 今日异常与预警
3. 执行总览

第二屏继续承接：
4. 协同域摘要
5. 快捷入口
6. 经营摘要（轻量）
```

## 6. 模块定义

### A. 我的待办

#### 目标

把“需要我动作”的事项放在首页最上层，减少用户登录后还要自己判断去哪个系统的成本。

#### 推荐卡片

- 待审批工单
- 待开工工序任务
- 待报工任务
- 待检验任务
- 待质量裁决
- 待异常处理
- 待版本切换确认

#### 静态字段

| 字段          | 说明                                     |
| ------------- | ---------------------------------------- |
| `id`          | 卡片唯一标识                             |
| `title`       | 卡片标题                                 |
| `count`       | 当前数量                                 |
| `severity`    | 视觉级别，如 `normal / warning / danger` |
| `ownerSystem` | 主责系统                                 |
| `targetRoute` | 点击跳转路由                             |
| `targetLabel` | 跳转入口名称                             |
| `helperText`  | 补充说明，例如“今日新增 2”               |

#### 推荐跳转

| 卡片           | 跳转                                 |
| -------------- | ------------------------------------ |
| 待审批工单     | `/mes/work-order/list`               |
| 待开工工序任务 | `/mes/execution/my-tasks`            |
| 待报工任务     | `/mes/execution/my-tasks`            |
| 待检验任务     | `/collaboration/qms/inspection`      |
| 待质量裁决     | `/collaboration/qms/ncr`             |
| 待异常处理     | `/mes/traceability/exception-center` |
| 待版本切换确认 | `/engineering-plan/plm/cutover`      |

### B. 今日异常与预警

#### 目标

把“阻塞执行的风险”从普通摘要中单独拉出来，优先暴露。

#### 推荐列表

- 缺料预警
- 设备停机 / 采集告警
- 检验失败待处置
- 冻结 WIP
- 超期工单
- APS / MRP 例外

#### 静态字段

| 字段             | 说明                      |
| ---------------- | ------------------------- |
| `id`             | 预警项标识                |
| `type`           | 预警类型                  |
| `title`          | 预警标题                  |
| `level`          | `warning / danger / info` |
| `count`          | 数量                      |
| `ownerSystem`    | 主责系统                  |
| `affectedObject` | 影响对象                  |
| `actionText`     | 处理动作文案              |
| `targetRoute`    | 处理页跳转                |

### C. 执行总览

#### 目标

让用户快速知道执行面今天在发生什么，而不是只看到经营数据。

#### 推荐摘要卡

- 今日开工工单
- 生产中工单
- 生产中工序任务
- 今日报工数量
- 待检 WIP
- 返工中数量

#### 推荐可视化

- 工单状态分布
- 工序执行漏斗
- 车间负荷摘要

#### 静态字段

| 字段          | 说明            |
| ------------- | --------------- |
| `id`          | 指标标识        |
| `title`       | 指标名称        |
| `value`       | 指标值          |
| `unit`        | 单位            |
| `trend`       | 与昨日/本周对比 |
| `targetRoute` | 跳转页          |
| `ownerSystem` | 主责系统        |

### D. 协同域摘要

#### 目标

让用户看到哪些外围系统正在影响执行中枢，而不必先进入各自菜单。

#### 推荐域卡片

- `WMS` 仓储与批次
- `QMS` 质量裁决
- `设备 / IoT`
- `工程计划`
- `客户交付`

#### 每张域卡片建议摘要

| 协同域       | 建议摘要                     |
| ------------ | ---------------------------- |
| `WMS`        | 低库存、待领料、待入库       |
| `QMS`        | 待检、失败、待复检关闭       |
| `设备 / IoT` | 运行、停机、告警             |
| `工程计划`   | 待齐套、待切换、扰动例外     |
| `客户交付`   | 今日交付、延期风险、变更订单 |

#### 静态字段

| 字段           | 说明           |
| -------------- | -------------- |
| `domain`       | 协同域标识     |
| `title`        | 协同域标题     |
| `ownerSystem`  | 主责系统       |
| `summaryItems` | 摘要项数组     |
| `targetRoute`  | 协同域默认跳转 |

`summaryItems` 建议结构：

| 字段          | 说明     |
| ------------- | -------- |
| `label`       | 摘要名   |
| `value`       | 数值     |
| `level`       | 视觉级别 |
| `targetRoute` | 明细跳转 |

### E. 快捷入口

#### 目标

把静态阶段最常点击的高频页面放出来，减少从左侧菜单逐层找入口的成本。

#### 推荐入口

- 工单列表
- 我的任务
- 工序报工
- 在制品
- 检验任务
- 生产领料
- 异常处置
- 生产看板
- 排程结果
- 产品结构

#### 静态字段

| 字段          | 说明     |
| ------------- | -------- |
| `id`          | 入口标识 |
| `title`       | 入口名称 |
| `icon`        | 图标标识 |
| `targetRoute` | 跳转路径 |
| `group`       | 所属域   |
| `ownerSystem` | 主责系统 |

### F. 经营摘要（轻量）

#### 目标

保留必要的健康度判断，但不让首页再次退化成经营图表页。

#### 推荐指标

- 订单交付率
- 设备 OEE
- 低库存风险数
- 超期工单数

#### 特别约束

- 首页只显示轻量摘要
- 不在首页展开长周期收入、成本、利润大图
- 完整经营分析统一引导到 `管理分析 / 经营驾驶舱`

#### 静态字段

| 字段          | 说明                     |
| ------------- | ------------------------ |
| `id`          | 指标标识                 |
| `title`       | 指标名称                 |
| `value`       | 指标值                   |
| `unit`        | 单位                     |
| `trend`       | 趋势值                   |
| `targetRoute` | 跳转到经营驾驶舱或专题页 |

## 7. 首页与经营驾驶舱的边界

### 工作台负责

- 待办
- 预警
- 执行态势
- 协同摘要
- 快捷入口
- 少量健康度摘要

### 经营驾驶舱负责

- 营收趋势
- 成本趋势
- 利润趋势
- 交付趋势
- 长周期经营分析
- 财务与经营主题下钻

### 统一原则

首页回答：

> 现在要做什么？

驾驶舱回答：

> 最近经营表现怎么样？

## 8. 静态数据建议结构

首页静态数据建议统一由一个聚合对象承接：

```ts
interface WorkbenchStaticData {
  todoCards: TodoCard[]
  alerts: AlertItem[]
  executionStats: ExecutionStat[]
  executionCharts: ExecutionChartGroup
  collaborationDomains: CollaborationDomainCard[]
  shortcuts: ShortcutItem[]
  businessSummary: SummaryMetric[]
}
```

说明：

- 第一版先只要求字段稳定，不要求真实计算逻辑
- 各模块可以先从现有 mock / static provider 聚合
- 后续接 mock 或 API 时，优先保持字段结构不变

---

## 9. 静态页面阶段实施顺序

### 第一阶段：先搭结构

- 固定首页六大模块
- 固定模块顺序
- 固定卡片层级

### 第二阶段：补静态数据

- 为每个模块补静态字段
- 用本地静态 provider 输出聚合对象

### 第三阶段：补跳转

- 每张卡片点击进入对应页面
- 快捷入口可直接打开目标页

### 第四阶段：补角色化

- 后续再做按角色裁剪
- 第一版不在首页里引入复杂权限差异

---

## 10. 首版不做什么

以下内容建议不在首页第一版静态重构中强行加入：

- 复杂审批流时间轴
- 大段经营图表墙
- 自由拖拽自定义布局
- 多角色完全不同的首页模板
- 实时 websocket 刷新逻辑

原因：

- 当前阶段目标是先稳定入口结构
- 不是先做复杂工作台平台能力

---

## 11. 直接指导前端重构的结论

首页静态重构时，请直接按以下原则执行：

1. 先删掉“经营驾驶舱式首页”的默认思路。
2. 首页优先展示待办、预警和执行摘要。
3. 首页所有模块都必须带明确跳转。
4. 首页只聚合，不维护对象真相。
5. 深度经营分析统一留给 `管理分析 / 经营驾驶舱`。
