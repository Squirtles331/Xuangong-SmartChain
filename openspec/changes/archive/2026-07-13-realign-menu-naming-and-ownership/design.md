## Context

当前仓库已经具备较完整的前端路由骨架，但菜单名称、页面归位和系统边界并未稳定收敛。

现状问题主要集中在三类：

1. 用户可见命名不友好  
   当前菜单大量使用系统缩写、平台视角和实现视角命名，普通产品用户难以快速理解入口用途。

2. 菜单分组与系统设计不完全一致  
   路由实现、菜单说明文档、项目总览和核心对象主责矩阵之间存在口径差异，例如 `MDM` 的层级定位、`版本切换` 的页面属性、`返工` 和 `异常` 的责任表达。

3. 菜单位置与责任归属混淆  
   页面放在哪个菜单里，并不等于该系统拥有对象真相；当前文档和页面表达容易让使用者误解 `WMS/MES`、`QMS/MES`、`PLM/MES` 等边界。

这次设计需要先把产品导航语言、架构分层语言和系统主责语言拆开，再重新建立可落地的统一基线。

## Goals / Non-Goals

**Goals:**

- 建立一套用户可理解、与架构一致的菜单信息架构。
- 为每个页面补齐主责系统、协同系统、核心对象和边界说明。
- 输出一份可直接驱动后续前端重构的菜单真相表。
- 为后续静态页面开发提供稳定入口，避免边做页面边改菜单口径。

**Non-Goals:**

- 本变更不直接实现页面代码重写。
- 本变更不直接实现 mock、接口或状态机细节。
- 本变更不强制一次性修改所有路由路径；首轮以稳定菜单语义和责任元数据为主。

## Decisions

### Decision 1: 用户菜单名称与系统内部模块名分离

用户看到的菜单 SHALL 采用业务对象或用户任务语言，而不是直接照搬系统缩写或工程目录名。

采用原因：

- 用户寻找的是“我要做什么”，不是“哪个系统负责什么”。
- 当前 `PLM / QMS / WMS / MDM` 等命名对实施、开发友好，但对产品用户不够直观。

替代方案：

- 直接保留现有系统缩写命名。  
  放弃原因：容易继续加深“系统菜单化”而不是“产品导航化”。

### Decision 2: 菜单分组以“用户任务流”优先，系统主责作为元数据表达

菜单分组以用户任务流和核心对象闭环为第一原则，系统主责作为页面元数据沉淀，而不是完全由菜单位置暗示。

推荐一级导航基线：

- `首页工作台`
- `生产执行`
- `工程计划`
- `质量仓储与协同`
- `客户与订单`
- `经营分析`
- `平台与主数据`

推荐二级分组基线：

- `生产执行`
  - `工单`
  - `工序与报工`
  - `WIP 与现场控制`
  - `追溯与异常`
- `工程计划`
  - `产品结构与工艺`
  - `计划排程与扰动`
- `质量仓储与协同`
  - `仓储与批次`
  - `质量裁决`
  - `采购与供应商`
  - `设备与采集`
- `客户与订单`
  - `客户经营`
  - `订单与交付`
- `经营分析`
  - `经营看板`
  - `成本与财务`
  - `综合分析`
- `平台与主数据`
  - `系统管理`
  - `主数据治理`
  - `平台设置`

替代方案：

- 严格按系统分层把所有菜单写成 `MES/PLM/WMS/QMS...`。  
  放弃原因：更利于架构表达，但不利于用户理解，也会把跨系统闭环切碎。

### Decision 3: 冲突裁决采用固定优先级

当“当前路由、菜单文档、项目总览、主责矩阵”出现冲突时，按以下优先级裁决：

1. `以MES为核心一体化平台整体业务流程.md`
2. `玄工智链_核心对象主责矩阵_V1.0.md`
3. `玄工智链_一体化平台开发参考总册_V1.0.md`
4. `玄工智链_系统菜单架构与页面说明_V1.0.md`
5. 当前 `src/router/index.ts`

原因：

- 先用平台业务逻辑和对象主责确认“谁负责什么”。
- 再用菜单文档和现有代码确认“现在放在哪里”。

### Decision 4: 重点冲突域必须显式记录“双重角色”

以下页面不允许只写单一归属，必须显式表达“发起/裁决/执行/真相”中的不同角色：

- `版本切换`：`PLM` 发起与版本合法性主责，`MES` 承接现场切换执行。
- `返工工单`：`QMS` 裁决是否返工，`MES` 负责返工链执行。
- `异常中心`：`MES` 管异常锁定和执行恢复，质量类结论由 `QMS` 决策。
- `投料 / 消耗`：`MES` 管投料与消耗事实，`WMS` 管库存移动和领退料真相。
- `MDM`：导航归入 `平台与主数据`，但必须标注其跨域治理属性，避免被误解成单纯平台设置。

替代方案：

- 只给页面定义一个系统归属。  
  放弃原因：会掩盖联合流程页面的真实责任结构。

### Decision 5: 首轮优先稳定名称、层级和元数据，不优先改路由路径

首轮输出以“统一命名、统一分组、统一责任字段”为主；已有路径在可行时保留，减少当前工程级联改动。

后续静态页面重构时，再按新菜单基线逐步处理：

- 路由标题
- 菜单显示名称
- 面包屑
- 占位页说明
- `activeMenu`
- 必要时的路径迁移与兼容

替代方案：

- 先彻底重命名路径和目录。  
  放弃原因：当前阶段高风险，且会把“命名问题”和“工程迁移问题”耦合。

### Decision 6: 工作台首页 SHALL 作为跨域行动工作台，而不是经营驾驶舱副本

`/` 对应的工作台首页必须重新定位为“登录后的统一行动入口”，而不是继续承担经营驾驶舱的大图表看板职责。

采用原因：

- 系统菜单文档已经把首页定义为“平台全局运行状态、待办任务、关键经营指标、快捷入口”的聚合页。
- 开发参考总册明确要求首页只做聚合，不维护任何对象真相。
- 当前首页内容与 `管理分析 / 经营驾驶舱` 高度重叠，造成“工作台”和“经营分析”定位混淆。
- 对制造执行型平台而言，首页的首要任务应该是告诉用户“现在该处理什么”，而不是先给出长周期经营分析图。

首页推荐结构基线：

- `我的待办`
- `今日异常与预警`
- `执行总览`
- `协同域摘要`
- `快捷入口`
- `经营摘要（轻量）`

各区块职责约束：

- `我的待办`：承接待审核工单、待开工任务、待报工、待检验、待裁决、不合格处理、待切换确认等行动事项。
- `今日异常与预警`：承接缺料、停机、质量失败、冻结批次、超期工单、计划例外等需要优先处理的阻塞信息。
- `执行总览`：承接工单、工序任务、WIP、报工、返工等执行对象的摘要态势。
- `协同域摘要`：承接 `WMS / QMS / 设备 / 工程计划 / 客户交付` 的摘要协同信息，但不在首页维护任何业务真相。
- `快捷入口`：承接工单列表、我的任务、报工、检验任务、生产领料、异常处置、排程结果等高频跳转。
- `经营摘要（轻量）`：仅保留少量摘要指标，并提供进入 `经营驾驶舱` 的入口，不在首页展开完整经营分析图组。

替代方案：

- 继续将首页做成“经营驾驶舱”大图表页。  
  放弃原因：会让首页与 `管理分析 / 经营驾驶舱` 重复，削弱工作台的任务入口价值。

- 完全移除经营指标，只保留待办和快捷入口。  
  放弃原因：首页仍需要少量跨域摘要，帮助用户快速判断平台运行健康度。

### Decision 7: 工作台首页内容必须围绕核心对象闭环组织，而不是围绕孤立指标组织

工作台首页的聚合块应优先围绕平台的高优先级闭环来组织：

`工单 -> 工序任务 -> WIP 批次 -> 检验结论 -> 异常锁定/恢复 -> 版本切换`

采用原因：

- 这条链路是当前平台开发参考总册中已明确的最高优先级闭环。
- 首页如果继续以“营收、成本、利润”一类孤立指标为主，会掩盖执行型平台最关键的阻塞和动作入口。

首页区块与核心对象的推荐映射：

| 首页区块       | 核心对象 / 结果域                                | 主要目标                         |
| -------------- | ------------------------------------------------ | -------------------------------- |
| 我的待办       | 工单、工序任务、检验任务、不合格单、版本切换任务 | 告诉用户下一步要处理什么         |
| 今日异常与预警 | 缺料、停机、冻结 WIP、计划扰动、质量失败         | 告诉用户当前哪里被阻塞           |
| 执行总览       | 工单、工序任务、WIP、报工、返工                  | 告诉用户执行面当前处于什么状态   |
| 协同域摘要     | WMS、QMS、设备、计划、客户交付                   | 告诉用户哪些外围系统正在影响执行 |
| 经营摘要       | 交付率、OEE、低库存、超期风险等摘要              | 告诉用户整体运行是否健康         |

### Decision 8: 工作台首页静态重构 SHALL 先做信息架构与跳转框架，再做个性化和深度分析

首页在静态页面阶段应按以下顺序推进：

1. 先稳定首页分区和模块顺序。
2. 再为每个分区准备静态聚合数据结构。
3. 再为每张卡片补齐跳转入口。
4. 最后才考虑按角色个性化、权限裁剪和深度下钻。

采用原因：

- 当前项目已经明确采用“先静态页面和静态数据，后 mock”的节奏。
- 首页属于入口页，必须先稳定结构，否则后续每补一个静态模块都可能重新打散布局和文案。

### Decision 9: 工作台首页 SHALL 提供可直接驱动静态页面开发的模块蓝图

工作台首页重设计不能只停留在“应该展示哪些内容”的概念层，还必须产出一份可直接驱动静态页面开发的模块蓝图。

蓝图至少要包含：

- 页面总线框结构
- 模块顺序与主次层级
- 每个模块的目标说明
- 每个模块的静态数据字段
- 每个模块的默认跳转关系
- 首页内容与 `经营驾驶舱` 的分工边界

采用原因：

- 当前项目后续要先做静态页面和静态数据，如果没有模块级蓝图，首页在实现时仍会边做边改。
- 首页是跨系统聚合页，模块多、跳转多、数据来源多，必须先把字段和入口稳定下来。

本变更下的首页模块蓝图参考文档：

- `homepage-workbench-blueprint.md`

## Risks / Trade-offs

- [风险] 新菜单名称与已有文档或团队习惯不一致 -> 缓解：在真相表中保留“旧称/别名/系统标签”字段，提供过渡期映射。
- [风险] 菜单更易懂后，开发人员反而看不出系统边界 -> 缓解：在页面元数据和文档中强制记录主责系统、协同系统和最终真相。
- [风险] 路由路径暂不调整会导致“名称新、路径旧” -> 缓解：首轮接受工程层兼容，第二轮在静态重构时统一清理。
- [风险] `MDM`、`版本切换` 等争议点继续反复 -> 缓解：在本变更里先形成书面裁决，不把争议留到页面实现阶段。
- [风险] 首页重设计后与现有经营驾驶舱职责分界不清 -> 缓解：规定首页只保留轻量经营摘要，完整经营趋势与经营图组下沉到 `管理分析 / 经营驾驶舱`。
- [风险] 首页聚合过多模块导致再次沦为“大杂烩” -> 缓解：强制首页所有模块都回答“它是在提示动作、提示阻塞，还是提示健康度”，不能只因为有数据就上首页。

## Migration Plan

1. 先形成菜单真相表，完成命名、层级、归属、冲突裁决。
2. 基于真相表修订菜单文档和前端开发计划。
3. 在前端路由元数据中补齐新菜单名称和责任字段。
4. 基于新的工作台定位，重写首页信息架构和首页与经营驾驶舱的分工边界。
5. 以新菜单基线重新排序“静态页面先行”的开发顺序。
6. 页面静态开发完成后，再进入 mock 和接口适配阶段。

回退策略：

- 若新命名反馈不佳，可保留旧称作为过渡别名。
- 若路径迁移带来风险，继续保留现路径，仅回退显示名称和分组。

## Open Questions

- `客户与订单` 是否保留当前文档中的 `客户与经营`，还是全面转向更直接的任务语言。
- `质量仓储与协同` 是否需要拆成两个一级导航，还是维持一个协同入口。
- `MDM` 在对外产品语言上是否单独强调“主数据平台”，还是继续作为 `平台与主数据` 的二级域。
- `经营分析` 下是否保留 `合规与综合管理` 这一分组名称，还是拆为更清晰的 `合规分析` 与 `人资安环`。
- 工作台首页在后续版本是否需要按角色提供不同视图，例如计划员、班组长、质量工程师、仓库人员的差异化工作台。
