## Context

根据当前仓库中的玄工开发基线，正式实施应遵循：

`MDM` 基础完成 -> `PLM` 定义链稳定 -> `MES` 执行链 -> `WMS/QMS`

已有文档已经明确：

- `工艺路线` 和 `工序` 属于 `PLM` 上游定义对象；
- `工序` 是工艺路线的子对象，不应被下游执行系统自行重定义；
- 静态页阶段必须先稳定页面结构、字段分组、状态标签和上下游关系区，再进入 mock。

当前代码中的主要偏差是：

- `PLM` 菜单下缺少“工序定义”主入口；
- 现有 `MDM/operation-master` 页面虽然路由元信息写成 `ownerSystem: PLM`，但页面标题、治理定位和访问入口仍然像 `MDM` 主数据页面；
- 页面本身缺少详情结构，无法承载“质量关卡、报工模式、关联工艺路线、影响 BOM 版本”等静态阶段应预留的信息区块；
- `工序编码`、`工艺路线编码` 目前缺少“规则归 MDM、实例归 PLM”的正式分责说明；
- `MDM` 菜单中仍保留 `BOM`、`工艺路线`、`工艺参数` 等可编辑维护页，会继续放大双入口认知偏差。

## Goals / Non-Goals

**Goals**

- 在 `PLM` 菜单下补齐工序定义静态页入口。
- 让页面结构符合静态阶段标准，包括详情和关联区。
- 明确 `PLM`、`MES`、`QMS`、`APS` 在工序对象上的边界。
- 明确 `MDM` 与 `PLM` 在工序编码、工艺路线编码上的规则与实例分责。
- 收敛 `MDM` 中与 `PLM` 主责冲突的维护入口，只保留治理型菜单。

**Non-Goals**

- 本次不引入真实接口或 mock 服务。
- 本次不重写工艺路线编辑器的数据模型。
- 本次不推进 `MES` 报工或 `QMS` 过程检验页面联调。

## Decisions

### Decision 1: 工序定义主入口迁移到 `PLM` 菜单

工序定义是 `PLM` 第一批次的工艺定义对象，用户应从 `PLM` 菜单进入，而不是从 `MDM` 菜单理解为纯治理目录。

原因：

- 与基线文档中 `PLM -> MES` 的实施顺序保持一致；
- 与 `master-data-object-catalog.md` 中“工序由 `PLM` 负责，供 `MES/QMS/APS` 消费”的结论一致；
- 可避免后续执行链页面继续把工序解释为本地字段集合。

### Decision 2: `MDM` 不保留工序定义的维护入口

本次不保留 `MDM` 工序定义兼容入口，也不在 `MDM` 菜单中继续暴露 `BOM`、工序、工艺路线、工艺参数实例的可编辑维护入口。

原因：

- 用户已经确认不需要兼容跳转；
- 保留双入口会继续强化“MDM 也能维护工序”的错误认知；
- 在对象主责已经明确的前提下，唯一可编辑入口应收敛到 `PLM`。

### Decision 5: `MDM` 只保留治理型页面，不保留工艺参数实例维护页

当前 `MDM/process-parameter` 页面包含新增、编辑、删除以及“适用工序”等实例字段，它属于工艺对象实例维护，而不是纯规则治理页，因此本次一并移除。

原因：

- 页面动作已经落入 `PLM` 主责边界；
- 参数实例需要跟随工艺路线、工序模板、版本与发布语义统一管理；
- 如果后续确有跨系统标准化参数模板需求，应以新的“规则/标准/基线”页面语义重建，而不是沿用当前实例维护页。

### Decision 3: 工序定义静态页必须补齐详情与关联区

工序定义页不仅要展示编码、名称和默认工作中心，还要预留静态阶段的详情结构，包括：

- 基础定义
- 工时与资源
- 质量与报工约束
- 关联工艺路线 / BOM 版本 / 变更单

原因：

- 满足静态页先稳定结构的基线要求；
- 为后续 `MES` 报工、`QMS` 过程检验、`APS` 排程提供统一字段形态；
- 避免 mock 阶段再临时新增第二套对象结构。

### Decision 4: 编码规则归 `MDM`，编码实例归 `PLM`

`工序编码` 与 `工艺路线编码` 的规则定义由 `MDM` 负责，包括前缀、位数、分段规则、是否带版本段、是否按组织或产品线分池；具体编码实例的生成、占用、唯一性校验、停用和版本发布由 `PLM` 负责。

原因：

- 编码规则属于跨对象、跨模块的全局治理能力，应放在 `MDM`；
- 编码实例属于具体对象的生命周期管理，应由对象拥有者 `PLM` 负责；
- 这样既能保持全局编码规则统一，又不会让 `MDM` 越权承担工艺对象维护责任。

## Risks / Trade-offs

- 风险：移除 `MDM` 维护入口后，历史使用习惯需要调整。  
  缓解：在 `PLM` 页面和变更文档中明确唯一维护入口，并在评审时同步对象主责。

- 风险：本次只做静态页，后续仍需补充 mock 和接口映射。  
  缓解：在详情结构中提前固定字段区块，后续直接继承。

## Migration Plan

1. 创建新的 OpenSpec 变更文档，锁定范围为 `PLM` 工序定义静态页和编码分责。
2. 在 `PLM` 路由中新增工序定义主入口。
3. 从 `MDM` 菜单中移除 `BOM`、工序、工艺路线、工艺参数实例维护入口，并只保留编码规则治理职责。
4. 用新的静态页实现替代当前“MDM 风格”的工序页面表达。
5. 在开发时同步落实“规则归 MDM，实例归 PLM”的编码约束。
