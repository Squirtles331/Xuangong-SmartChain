## Context

`ERP` 首批静态页已经把 `需求池 -> MRP 结果链 -> 总账对账` 这条主链稳定下来，但当前仓库里剩余的 `ERP` 页面仍然处于“已有页面、尚未正式冻结静态边界”的中间态：

- `预测需求` 仍沿用旧 `MRP` API 语义，页面既像录入页，又混入了趋势演示逻辑
- `多工厂计划` 仍直接依赖旧 `MRP` API，缺少清晰的供需平衡对象边界
- `应付管理`、`成本分析`、`财务报表` 仍依赖旧 `finance` API 口径，页面文案和对象归属也还未收口
- 第二批页面尚未纳入统一的 `ERP` 静态服务层，后续 mock/API 若直接继承当前结构，容易把旧字段和旧边界继续固化

这一批的目标不是进入真实接口联调，而是先把 `ERP` 余下扩展页在静态阶段按对象边界正式冻结，形成可以继续往 mock/API 继承的稳定页面结构。

## Goals / Non-Goals

**Goals:**

- 冻结 `ERP` 第二批页面的正式范围与推荐顺序
- 统一第二批页面到 `src/static/services/erp.ts`
- 冻结 `预测需求`、`多工厂计划`、`应付管理`、`成本分析`、`财务报表` 的主对象边界
- 保持第二批页面仍处于静态阶段，不引入真实 API 依赖
- 让后续 `SRM`、`CRM`、更深财务分析或 BI 扩展可以继承同一套静态字段和页面壳层

**Non-Goals:**

- 本次变更不引入真实预测算法、多工厂优化算法或财务核算引擎
- 本次变更不进入真实 API 联调或 mock/API 迁移阶段
- 本次变更不改写首批 `ERP` 页面业务语义
- 本次变更不将 `财务报表` 扩展为 `BI` 驾驶舱，不将 `应付管理` 扩展为资金执行页

## Decisions

### Decision 1: ERP 第二批范围固定为“预测 + 多工厂 + 应付 + 成本 + 报表”

本次正式纳入实现的页面固定为：

- `预测需求`
- `多工厂计划`
- `应付管理`
- `成本分析`
- `财务报表`

这样可以把 `ERP` 菜单内剩余的既有页面全部完成静态收口，同时又不越界到 `SRM`、`CRM` 或 `BI`。

备选方案是只先做 `预测需求` 和 `应付管理`，把分析页继续后置；但这样会让 `ERP` 菜单长期保留一半旧页、一半新页，口径继续分裂，因此不选。

### Decision 2: 第二批页面继续复用同一 ERP 静态服务层

第二批页面不单独创建新的页面级假数据，而是继续扩展 `src/static/services/erp.ts`：

- 将预测、跨工厂平衡、应付、成本、财务报表都整理进同一服务层
- 由页面消费统一静态结构，而不是直接依赖旧 `@/api/mrp` 或 `@/api/finance`
- 让后续 mock/API 继续继承第二批字段结构

备选方案是给第二批页面分别保留各自旧 API 与 mock 逻辑，但这会破坏首批已经建立的 `ERP` 静态基线，因此不选。

### Decision 3: 预测与多工厂页面仍属于 ERP 计划扩展页，不回写 CRM / APS / WMS 真相

- `预测需求` 的主对象固定为 `ERP` 预测需求承接对象，允许显示来源类型和概率，但不得把它包装成客户需求真相页
- `多工厂计划` 的主对象固定为跨工厂供需平衡建议，允许显示工厂负荷、调拨/采购/生产建议，但不得把它包装成 APS 排程结果页或 WMS 库存台账页

备选方案是把这两页做成“协同总览页”；但这样会让主对象失焦，后续很难稳定字段边界，因此不选。

### Decision 4: 财务第二批页面按“承接页 -> 结果页 -> 聚合页”区分，而不是一个大看板

- `应付管理` 是承接对象页，使用共享 CRUD 容器
- `成本分析` 是结果聚合页，保留筛选、摘要与成本构成表达
- `财务报表` 是经营报表聚合页，保留月份对比、趋势与结构性指标，但不扩成 BI 大屏

备选方案是把这 3 页统一压成同一种财务概览页，但会模糊对象边界，因此不选。

## Risks / Trade-offs

- [Risk] 第二批页面中 `预测需求`、`财务报表` 原有页面都含有旧演示图表逻辑，重写时容易留下旧 API 或旧命名残余
  - Mitigation: 整页切换为 `ERP` 静态服务驱动，同时同步清理旧 API import
- [Risk] `成本分析` 与 `财务报表` 都属于分析/聚合层，若表达过重会重新演变成 BI 页
  - Mitigation: 保持摘要卡片 + 查询 + 结果块的轻量结构，不引入驾驶舱式布局
- [Risk] `应付管理` 很容易与 `SRM` 采购协同或资金执行混淆
  - Mitigation: 明确页面文案与字段命名只表达 `ERP` 应付承接，不回写供应商协同或付款执行

## Migration Plan

2. 冻结第二批页面范围、对象边界和静态服务结构
3. 扩展 `src/static/services/erp.ts`，补齐第二批页面静态数据
4. 重写第二批页面与表单弹窗，切到共享静态基线
5. 回填任务状态并做静态阶段验证
