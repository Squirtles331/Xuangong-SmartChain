# 玄工智链_核心事件清单_V1.0

## 一、文档目的

本文档用于把当前平台“事件驱动 + 状态控制 + 审批放行”的总体原则，落成可以实施的事件体系。

目标不是简单列消息名称，而是明确：

- 什么情况下触发事件
- 事件由谁发出
- 事件影响哪些对象
- 事件要驱动哪些下游动作
- 事件失败时如何重试、补偿、对账

---

## 二、事件设计原则

### 1. 事件描述“事实已经发生”

事件名称应表达已发生事实，而不是未发生意图。

示例：

- 用 `work_order.released`
- 不用 `release_work_order`

### 2. 一个事件只表达一个核心事实

不要把“工单下达 + 齐套通过 + 开工放行”打成一个事件。

### 3. 事件要能追溯到对象

每个事件必须能定位：

- 来源系统
- 业务对象类型
- 业务对象 ID
- 事件发生时间
- 事件前后状态

### 4. 事件驱动不等于强同步耦合

允许异步，但不能丢事实，不能重复污染状态。

---

## 三、标准事件头定义

后续所有事件建议至少带以下字段：

| 字段 | 说明 |
| --- | --- |
| `event_id` | 事件唯一编号 |
| `event_type` | 事件类型编码 |
| `source_system` | 来源系统 |
| `object_type` | 对象类型 |
| `object_id` | 对象主键 |
| `occurred_at` | 业务发生时间 |
| `published_at` | 事件发布时间 |
| `status_before` | 变更前状态 |
| `status_after` | 变更后状态 |
| `operator_id` | 操作人 / 系统 |
| `trace_id` | 跨系统追踪号 |
| `version` | 事件版本号 |

## 五、关键事件链示例

## 5.1 工单下达到开工

```text
work_order.created
-> material.kitting_verified
-> work_order.released
-> operation_task.ready
-> operation_task.started
```

## 5.2 质量异常到返工恢复

```text
inspection.result_failed
-> ncr.created
-> wip.batch_frozen
-> rework.decided
-> reinspection.closed
-> wip.batch_released
```

## 5.3 工程变更到现场切换

```text
eco.approved
-> bom.version_released / routing.version_released
-> cutover.requested
-> cutover.approved
-> cutover.completed
```

## 5.4 设备停机到恢复执行

```text
equipment.alarm_triggered
-> equipment.downtime_started
-> operation_task.paused
-> equipment.recovered
-> exception.unlock_approved
-> operation_task.ready / started
```

## 六、事件消费规则

所有下游消费方统一遵守以下规则：

1. 必须支持幂等消费。
2. 必须记录消费结果和失败原因。
3. 对关键事件必须支持重试。
4. 对关键状态变更事件必须保留审计链。
5. 消费失败不能静默吞掉。

---

## 七、事件优先级建议

| 优先级 | 适用事件 | 要求 |
| --- | --- | --- |
| `P1` | 工单下达、质量失败、停机、切换完成、审批通过 | 必须可追溯、可重试、可告警 |
| `P2` | 报工、入库、领料、检验合格 | 必须幂等、建议准实时 |
| `P3` | 看板刷新、报表同步、统计聚合 | 可异步批处理 |

## 八、结论

事件体系不是消息清单，而是平台运行机制本身。

后续所有接口、审批流、状态机、对账机制，都应回到本清单判断：

- 这是不是一个已发生事实
- 这个事实是否已经具备唯一来源
- 下游是否知道应该如何安全消费它

只有这三点稳定，平台的事件驱动才不会演变成“消息很多、状态更乱”。
