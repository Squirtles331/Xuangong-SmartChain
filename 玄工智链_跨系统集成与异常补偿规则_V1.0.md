# 玄工智链\_跨系统集成与异常补偿规则\_V1.0

## 一、文档目的

本文档用于定义玄工智链平台跨系统协同的集成方式、失败处理方式、补偿机制和对账规则。

它重点解决两个问题：

1. 业务上谁和谁集成、怎么集成
2. 集成失败后如何保证不丢事实、不乱状态、可追溯恢复

---

## 二、总体集成原则

### 1. 主责系统输出真相，协同系统消费真相

跨系统同步不能变成“相互改写同一对象”。

### 2. 优先事件驱动，必要时查询补足

- 状态变化类：优先事件驱动
- 明细读取类：允许查询拉取
- 大批量分析类：允许批处理同步

### 3. 关键动作必须幂等

至少以下动作必须幂等：

- 工单下达
- 报工入账
- 领料完成通知
- 检验结论同步
- 版本切换结果同步
- 停机事件同步

### 4. 集成失败不能靠人工猜测

必须能够知道：

- 哪条数据失败
- 失败在哪一跳
- 是否已重试
- 是否需要人工补偿

## 三、推荐集成模式

| 集成类型     | 推荐方式                         | 适用场景                     |
| ------------ | -------------------------------- | ---------------------------- |
| 主数据同步   | 主数据发布 + 增量同步 + 周期对账 | 物料、组织、工作中心、供应商 |
| 业务状态同步 | 事件驱动                         | 工单、检验结论、停机、切换   |
| 统计分析同步 | 批处理 / 数据服务                | 报表、经营看板、成本分析     |
| 实时采集     | 流式采集 / 事件上报              | IoT、设备告警、自动报工      |
| 补充查询     | API 查询                         | 详情页、校验页、追溯页       |

| -------- | ----------------------- | ---------------------------- | -------------------------- | ------------------- | -------------------- |

| `IoT`    | `MES`                   | 停机、告警、自动报工         | 驱动暂停、锁定、自动上报   | 实时事件            | `设备 / IoT`         |
| `MES`    | `ERP`                   | 完工、消耗、成本接口数据     | 成本归集与财务口径输入     | 批量 / 事件         | `ERP` 负责财务真相   |

## 五、典型协同闭环与补偿规则

## 5.1 销售订单变更闭环

```text
CRM 订单变更
-> ERP / APS 接收变化
-> MES 识别受影响工单与工序
-> 形成扰动分析
-> 执行局部调整或全局重排
```

补偿要求：

- 若 `APS` 未成功接收订单变化，必须进入“待重发”队列
- 若 `MES` 未成功识别影响对象，必须生成人工复核任务
- 同一订单变更重复推送时，消费方必须按 `order_id + version` 幂等处理

## 5.2 版本切换闭环

```text
PLM 发布新版本
-> MES 识别受影响工单 / WIP
-> 审批放行
-> 现场切换
-> 切换结果回写追踪
```

补偿要求：

- 若 `PLM` 已发布但 `MES` 未接收到版本事件，必须通过版本差异对账补发
- 若 `MES` 已部分切换但未完成收口，状态必须停留在 `partially_completed`
- 任何切换失败都不能直接回写为“已完成”

## 5.3 领料与消耗闭环

```text
MES 发起领料需求
-> WMS 完成出库配送
-> MES 确认投料与消耗
-> ERP / 成本分析接收消耗结果
```

补偿要求：

- `WMS` 领料成功但 `MES` 未确认收到时，不能自动视为已消耗
- `MES` 消耗成功但成本接口失败时，应允许后补入账，不回滚现场消耗事实
- 领料和消耗必须按不同对象编号分开对账

## 5.4 质量异常闭环

```text
QMS 判不合格
-> MES 锁定 WIP / 工序
-> WMS 隔离库存批次
-> QMS 裁决返工 / 特采 / 报废
-> MES / WMS 执行处置
```

补偿要求：

- 若 `QMS` 判定失败但 `MES` 未锁定成功，必须立即告警，且禁止对象继续流转
- 若 `WMS` 隔离动作失败，不得视为异常闭环完成
- 质量裁决与库存隔离必须建立统一 `trace_id`

## 5.5 设备停机闭环

```text
IoT 上报告警 / 停机
-> MES 暂停受影响工序
-> 设备系统维修处理
-> 恢复事件回传
-> 规则恢复或审批恢复
```

补偿要求：

- 若 `IoT` 停机事件丢失，必须通过设备停机日志与 `MES` 暂停记录日对账发现
- 自动恢复必须区分一般停机与关键停机
- 关键停机恢复后应进入审批解锁，而不是直接继续执行

## 六、异常补偿机制

## 6.1 重试规则

| 场景         | 建议策略               |
| ------------ | ---------------------- |
| 临时网络失败 | 自动重试               |
| 对方系统超时 | 指数退避重试           |
| 幂等冲突     | 记录并人工复核         |
| 主数据缺失   | 暂停消费并进入补数队列 |
| 业务规则冲突 | 不自动重试，转人工处理 |

建议重试策略：

- 第 1 次：立即重试
- 第 2 次：5 秒后
- 第 3 次：30 秒后
- 第 4 次：5 分钟后
- 超过阈值：进入死信或补偿池

## 6.2 死信 / 补偿池

所有关键失败消息建议进入统一补偿池，至少包含：

- 消息编号
- 来源系统
- 目标系统
- 对象类型
- 对象编号
- 失败时间
- 重试次数
- 失败原因
- 当前处理状态

补偿池状态建议：

- `pending_retry`
- `waiting_manual_fix`
- `replayed`
- `closed`

## 6.3 人工补偿原则

人工补偿不是直接改数据库，而是：

1. 先定位主责系统真相
2. 再重放事件或补录业务动作
3. 最后留痕关闭补偿任务

---

## 七、幂等与唯一键建议

| 业务动作         | 幂等键建议                            |
| ---------------- | ------------------------------------- |
| 工单下达         | `work_order_id + release_version`     |
| 报工提交         | `operation_task_id + report_no`       |
| 领料完成         | `issue_order_id + line_no`            |
| 消耗入账         | `consumption_id`                      |
| 检验结果同步     | `inspection_task_id + result_version` |
| 停机事件同步     | `equipment_event_id`                  |
| 版本切换结果同步 | `cutover_task_id + step_no`           |

## 八、对账机制

## 8.1 日常对账清单

| 对账主题        | 对账双方             | 核心检查项                       |
| --------------- | -------------------- | -------------------------------- |
| 订单变更对账    | `CRM` vs `ERP / APS` | 订单版本、数量、交期             |
| 工单承接对账    | `APS` vs `MES`       | 已发布排程是否都被承接           |
| 领料 / 消耗对账 | `WMS` vs `MES`       | 领料数量、消耗数量、退料差异     |
| 检验 / 锁定对账 | `QMS` vs `MES / WMS` | 判退对象是否已冻结 / 隔离        |
| 版本切换对账    | `PLM` vs `MES`       | 已发布版本与现场生效版本是否一致 |
| 停机影响对账    | `IoT` vs `MES`       | 停机事件是否映射到受影响任务     |

## 8.2 对账频率建议

- 关键执行对象：准实时 + 日终对账
- 主数据与版本对象：发布后即时对账 + 日终巡检
- 分析统计类：T+1 对账即可

---

## 九、监控与告警建议

建议建立以下集成监控指标：

1. 事件发布成功率
2. 事件消费成功率
3. 死信数量
4. 重试次数分布
5. 对账差异数量
6. 关键对象锁定未同步数量
7. 版本切换未收口对象数量

高优先级告警建议包括：

- `QMS` 已判不合格但 `MES` 未锁定
- `WMS` 已隔离失败
- `MES` 消耗已入账但成本接口长时间未同步
- `PLM` 新版本已发布但 `MES` 未识别
- `IoT` 停机已上报但工序未暂停

---

## 十、实施落地建议

后续真正建设时，建议把跨系统集成拆成四层：

1. `对象层`
   定义对象主责和唯一键
2. `事件层`
   定义事件事实与消费动作
3. `补偿层`
   定义失败重试、死信、人工补偿
4. `对账层`
   定义日常校验和问题闭环

---

## 十一、结论

平台是否稳定，往往不取决于页面做得多快，而取决于：

- 系统边界是不是稳定
- 集成失败后有没有补偿机制
- 关键对象能不能对账

只要补偿、幂等、对账三件事定住，这套一体化平台就不容易在后期扩展时失控。
